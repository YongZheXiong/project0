# DEC-005：迭代一语义导航采用 YAML 语义位置表 + 规则匹配

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0的 P5 阶段需要实现基础语义导航，其边界是“语义输入 → 理解 → 导航目标转换 → 结果反馈”，导航执行本身归属于能力4。[13]

迭代一功能清单明确：语义位置管理采用 YAML 语义位置表文件，用于维护语义标签与坐标映射；语义输入解析采用 ROS2 服务/话题接口 + 字符串精确匹配。[13]

迭代一到迭代二技术升级路径明确：语义导航在 YAML 表基础上叠加 LLM 增强或语义地图，YAML 表保留为基础结构，不需要推翻重来。[3]

## 决策

项目0迭代一语义导航采用：

> YAML 语义位置表 + 规则匹配。

具体约定：

1. 使用 YAML 文件维护语义标签与地图坐标/区域之间的映射关系。
2. 语义输入解析采用确定性的字符串精确匹配或规则匹配。
3. 解析结果转换为导航目标，再交由 Nav2 执行。
4. 迭代一不引入 LLM 或复杂自然语言理解作为主链路。
5. 迭代二可在 YAML 表基础上叠加更强语义解析能力。

## 理由

1. 迭代一语义导航的目标是先跑通“语义输入到导航目标”的确定性链路，而不是直接做复杂语义理解研究。[13]
2. YAML 语义位置表简单、透明、可编辑、易调试，适合单人开发和真机验证。
3. 迭代二升级路径已经明确 YAML 表会保留为基础结构，可在其上叠加 LLM 增强或语义地图，不需要推翻重来。[3]
4. 该方案便于后续在报告和答辩中清楚解释语义导航链路：语义标签如何定义、如何转换为坐标、如何触发导航执行。[14]

## 影响

- 影响阶段：P5、P6、P9。
- 影响模块：p0_map_manager、p0_semantic_nav、p0_task_manager、p0_navigation。
- 影响文件：maps 或 config/semantic 下的语义位置表。
- 影响接口：语义目标查询、语义导航 action/service、任务状态反馈。
- 影响验收：语义目标识别、无效目标反馈、语义目标导航成功率。

## 回退条件（可选）

如果规则匹配无法覆盖迭代一验收所需语义输入，应优先收敛语义输入格式，而不是提前引入 LLM；只有在迭代二阶段且基础链路稳定后，再评估增强解析能力。
