# DEC-010：迭代一底盘采用四轮驱动差速式地面移动平台

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0定位为面向教学楼走廊真实场景、以真机闭环为验收标准、具备工程平台属性与科研实验承载能力的具身智能地面移动平台。

项目0迭代一的目标是在真实小车平台上完成“感知—决策—执行”闭环，并形成具备基本运动控制、SLAM导航、语义导航、安全约束、运行管理、数据记录、异常处理和人工接管能力的类产品级基础平台。

P2 阶段的本质是让小车在物理世界中能动、能停、能被人工接管，建立后续所有能力的物理前提；P2 需要完成底盘机械组装、电机驱动、编码器读取、PID 速度闭环、通信协议、里程计发布、遥控接管、急停测试和真实场地基础运动验证。

系统总架构中，执行层负责接收速度指令、驱动电机执行真实运动，并向上反馈轮速、里程计与底层异常状态；同时，执行层依赖底盘物理结构，具体由硬件架构文档定义。

软件架构中，上层到底盘执行层的控制接口已经抽象为 `/p0/base/cmd_vel`，底盘向上反馈 `/p0/base/odom`、`/p0/base/state` 和 `/p0/base/mode_feedback`，因此上层软件主链路不应直接耦合具体车轮数量与电机控制细节。

## 决策

项目0迭代一底盘采用：

> 四轮驱动的差速式地面移动平台。

具体约定：

1. 底盘物理层面采用四轮驱动结构。
2. 系统控制抽象上按差速式地面移动平台处理。
3. 上位机通过标准速度指令表达底盘控制意图，即线速度与角速度。
4. 底层控制器负责将上位机控制意图转换为左右侧驱动执行。
5. 对 ROS2 上层模块保持统一底盘接口：`/p0/base/cmd_vel`、`/p0/base/odom`、`/p0/base/state`、`/p0/base/mode_feedback`。
6. 迭代一阶段不展开四轮独立高级运动控制，不将底盘控制复杂化为独立研究课题。
7. 若后续需要利用四轮独立编码器信息、轮间补偿或更复杂底盘运动模型，放入迭代二或后续增强阶段评估。

## 理由

1. 项目0主线是地面移动具身智能平台，四轮驱动底盘符合项目0“地面移动平台”的总体方向，不涉及机械臂、多机器人、无人机、人形等非主线方向。
2. 迭代一首要目标是平台成立和真实闭环成立，因此底盘应优先保证稳定运动、可靠停车、人工接管、里程计反馈和基础安全，而不是在 P2 阶段引入过高复杂度的四轮独立高级控制。
3. 差速式控制抽象能够让上层导航、定位、任务调度、安全管理继续使用统一的速度指令、里程计和底盘状态接口，避免软件架构因底盘物理结构变化而大规模重构。
4. 系统总架构明确执行层负责真实运动执行与底层状态反馈，但具体底盘物理结构由硬件架构文档承接，因此四轮驱动应作为硬件架构与底层控制实现细节补充，而不是推翻系统总架构。
5. 项目0强调单人 solo 开发、阶段式推进、可执行、可验收、可回退，因此将四轮驱动约束为“差速式统一接口”更符合单人可控性和 P2-P6 连续推进需求。
6. 该决策有利于保持项目0的平台属性：底层可使用四轮驱动提升牵引与稳定性，上层仍保持可复现、可扩展、可调试的软件骨架，为后续迭代二增强保留空间。

## 影响

- 影响阶段：P1、P2，并影响后续 P3-P6 真机调试。
- 影响模块：执行层、p0_base_bridge、p0_safety_manager、p0_system_manager、p0_data_tools。
- 影响硬件文档：需要在硬件架构中明确四轮驱动底盘结构、电机数量、编码器接入、左右侧电机控制关系和轮距/轴距参数。
- 影响协议文档：需要在底盘通信协议中明确上位机下发速度指令、底层反馈里程计与底盘状态；如上传四轮原始编码器数据，应作为底层诊断或扩展字段处理。
- 影响接口文档：`/p0/base/odom` 保持标准里程计抽象，`/p0/base/state` 可保留四轮编码器、电机状态或左右侧聚合速度等诊断字段。
- 影响验收：P2 需要验证四轮驱动底盘的前进、后退、转向、停止、速度响应、通信中断停车、急停和人工接管能力。

## 回退条件（可选）

如果四轮独立控制、四路编码器闭环或轮间同步在 P2 阶段导致调试成本过高，应回退到更简单的左右侧聚合控制方式，即左侧电机组与右侧电机组分别形成目标速度闭环。

如果四轮驱动在真实走廊场景中出现明显打滑、转向不稳定或里程计误差过大，应优先通过机械结构检查、速度限幅、PID 调参、轮距/轴距参数修正和低速测试解决；在迭代一阶段不优先引入复杂运动学补偿或高级控制算法。
