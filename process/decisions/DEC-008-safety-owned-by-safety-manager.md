# DEC-008：安全判断与保护处置集中于 p0_safety_manager

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0采用三层安全结构：硬件急停不经过软件，STM32 负责限幅和超时停车，Orin NX 上的安全监控节点负责上层监测与处置。[18]

系统总架构中，安全保障模块负责风险约束、异常监测、安全处置和人工接管，并且可中断主链路任意模块。[20]

软件架构明确指出，不能把安全检测散落到所有包里各自处理；真正的安全判断与处置仲裁必须由 p0_safety_manager 集中承担，否则后续会出现多处急停逻辑打架。[15]

## 决策

项目0中安全判断与保护处置集中归属：

> p0_safety_manager。

具体约定：

1. p0_safety_manager 负责上层安全监测、安全状态发布、软件急停和保护处置。
2. 各业务模块可以输出自身状态或异常信息，但不各自独立执行最终安全仲裁。
3. 硬件急停保持独立，直接切断电机驱动供电，不经过软件链路。
4. STM32 负责底层限幅、超时停车和底层保护。
5. p0_safety_manager 与 p0_system_manager 协同处理系统级安全模式。

## 理由

1. 安全保障模块在系统总架构中可中断主链路任意模块，因此必须有清晰、集中的安全仲裁边界。[20]
2. 计算与通信架构已经明确三层安全结构：硬件急停、STM32 底层保护、Orin NX 安全监控节点。[18]
3. 软件架构明确禁止把安全检测散落到所有包里各自处理，否则急停逻辑会冲突。[15]
4. 集中式安全管理有利于异常记录、rosbag 回放、问题复盘、P6 安全测试和最终答辩表达。[5][14]

## 影响

- 影响阶段：P2-P6 全过程。
- 影响模块：p0_safety_manager、p0_base_bridge、p0_system_manager、p0_navigation、p0_task_manager。
- 影响接口：安全状态、安全事件、急停命令、底盘状态、定位/导航异常状态。
- 影响验收：急停响应、通信中断停车、导航异常停车、人工接管可靠性。

## 回退条件（可选）

如果 p0_safety_manager 初期实现复杂，应先实现最小安全闭环：急停状态、通信异常、底盘异常、导航异常停车；后续再扩展更多监控项。不得将最终安全仲裁拆散到多个业务包中。
