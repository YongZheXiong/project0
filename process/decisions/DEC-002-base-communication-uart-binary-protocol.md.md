# DEC-002：底盘通信采用 UART + 自定义二进制协议帧

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0采用 Orin NX 作为主计算平台，STM32 作为底层控制平台；Orin NX 与 STM32 之间通过 UART 双向通信，Orin NX 下发速度指令，STM32 上传编码器与底层状态反馈。[11]

P1 阶段需要确定底层通信方案，因为它会直接影响 Orin NX 与 STM32 职责划分、底层协议文档、P2 底盘运动闭环、后续接口定义和联调方式。[9]

迭代一功能清单已经将底层通信定义为上位机与下位机双向通信、协议帧格式和异常检测，并明确采用 UART 串口 + 自定义二进制协议帧。[4]

## 决策

项目0迭代一底盘通信采用：

> UART 串口 + 自定义二进制协议帧。

具体约定：

1. Orin NX 通过 UART 向 STM32 下发底盘运动控制意图。
2. STM32 通过 UART 向 Orin NX 上传编码器、底盘状态和异常状态。
3. 协议帧采用自定义二进制格式。
4. 迭代一实现超时检测、校验失败检测和通信中断自动停车。
5. 迭代二如需扩展底盘数据，不更换通信方式，只扩展协议帧字段。

## 理由

1. 当前计算与通信架构已经明确 Orin NX 与 STM32 之间通过 UART 进行速度指令下发和编码器/状态上传。[11]
2. 迭代一功能清单已明确底层通信采用 UART 串口 + 自定义二进制协议帧，并要求支持异常检测。[4]
3. 迭代一到迭代二技术升级路径明确：底层通信方案不变，如需扩展数据则扩展帧字段，不需要推翻重来。[3]
4. 该方案对单人开发更可控，能优先支撑 P2 底盘运动闭环和底层安全保护成立。

## 影响

- 影响阶段：P1、P2，并影响后续 P3-P6 真机联调。
- 影响模块：p0_base_bridge、STM32 固件、p0_safety_manager、p0_system_manager。
- 影响文档：计算与通信架构、底盘通信协议文档、接口定义文档、P2 实施文档。
- 影响验收：底盘速度控制、编码器反馈、通信异常停车、急停与人工接管测试。

## 回退条件（可选）

如果 UART 在真实平台上出现不可接受的稳定性问题，且通过波特率调整、校验机制、协议重发/超时机制仍无法解决，则重新评估通信方案。但在迭代一阶段，优先修正协议与实现，不轻易更换通信主路线。
