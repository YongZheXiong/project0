# DEC-011：项目0版本号规范

- 状态：确认
- 阶段：P1
- 日期：2026/4/24

## 背景
项目0已经进入需要持续沉淀文档、代码、配置、架构设计和阶段输出的状态。当前仓库中已经存在 README、CHANGELOG，以及与项目推进阶段对应的版本线路图；其中版本线路图已经明确给出：

- V0.x.x：对应 P0-P1，表示项目定义收束、技术选型、系统架构设计与工程准备完成
- V1.x.x：对应 P2-P6，表示迭代一完成，最小闭环正式成立
- V2.x.x：对应 P7-P10，表示迭代二完成，形成正式升级版本

与此同时，CHANGELOG 中已经开始采用 `0.1.0`、`0.1.1` 这样的三段式版本写法，README 也已经记录了当前版本号与最近更新状态。

如果现在不尽快统一版本号规范，后续会出现以下问题：

1. 文档版本、代码版本、Git 标签、阶段成果版本之间难以对应。
2. README、CHANGELOG、阶段报告、演示材料可能各自使用不同写法，造成表达混乱。
3. 单人开发过程中，难以快速判断某个版本究竟处于 P0/P1、迭代一、还是迭代二。
4. 后续如果要做阶段归档、实验对比、复试展示或答辩表达，版本历史会缺乏清晰的叙事结构。
5. 若不提前规范，随着 P2-P10 推进，版本号体系会越来越难以回收整理。

因此，需要在 P1 阶段就正式确定项目0的版本号规范，使仓库版本、阶段版本和对外表达版本形成统一映射关系。

## 决策
采用[主版本号:次版本号:修订号].
主版本号:
    0:P0-P1 项目准备阶段
    1:P2-P6 迭代一
    2:P7-P10 迭代二
次版本号:
    分别与阶段相对应.从1开始.例如,对于主版本号0,只有0.1.x和0.2.x,没有0.3.x等等.
修订号:
    当前阶段有修补,新增等更改,则增加修定号.从0开始
