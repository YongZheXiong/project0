# DEC-001：迭代一总体技术路线遵循“增量增强，不推翻重来”

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0在 P1 阶段需要完成迭代一功能清单、技术选型、系统架构、工程环境和 WBS/Sprint 规划，这些内容会直接驱动后续系统架构设计、ROS2 节点规划、接口定义、硬件方案和工程环境搭建。[8][9]

项目0的版本路线中，V0.1 对应 P0-P1，表示项目定义收束、技术选型、系统架构设计与工程准备完成，项目进入可执行开发状态；V1.0 对应 P2-P6，表示迭代一完成并形成真实可运行、可复现、可扩展、可调试、可演示的具身智能小车基础平台。[6]

同时，项目0的系统总架构在迭代二阶段本身不变，重点变化发生在层内能力增强，而不是重新更换总架构骨架。[20]

## 决策

项目0迭代一总体技术路线遵循：

> 先建立可运行、可复现、可调试、可演示的基础平台；迭代二在既有骨架上做增量增强，不推翻重来。

具体约定：

1. 迭代一所有关键技术选型必须优先服务于最小闭环成立。
2. 迭代二只在既有系统总架构、软件架构和接口骨架上增强能力。
3. 不采用会导致后续整体推翻重构的迭代一方案。
4. 若某项高级能力当前会显著增加复杂度，则优先推迟到迭代二。

## 理由

1. 迭代一目标是让执行层、感知层、空间认知层、行动规划层、任务决策层以及安全、管理、记录三条辅助链路全部成立，核心是最小闭环可运行。[20]
2. 迭代一到迭代二的技术升级路径已经明确：底层通信、电机控制、SLAM、导航框架、语义导航、任务状态机、安全机制、数据记录、系统管理均为增量式增强，不存在推翻重来。[3]
3. 软件架构也明确说明，迭代二不推翻软件架构，只增强感知与空间认知、语义导航、数据与实验支撑等局部能力。[1]
4. 该决策有利于后续形成类论文式总报告、实验验证成果、答辩 PPT、演示视频和关键问题回答材料，因为技术路线连续、演进逻辑清晰。[2][14]

## 影响

- 影响阶段：P1-P10 全过程。
- 影响模块：所有主链路模块与安全、管理、记录三条辅助链路。
- 影响文档：系统架构、软件架构、功能清单与技术选型、版本路线、WBS/Sprint 规划。
- 影响后续表达：项目报告、答辩 PPT、项目复盘、导师沟通材料。

## 回退条件（可选）

如果某个迭代一选型在 P2-P6 中被证明无法支撑真实平台最小闭环，且多次局部修正仍无法解决，则允许回退到更简单、更稳定、更可控的方案，但回退必须不破坏项目0总架构主链路。
