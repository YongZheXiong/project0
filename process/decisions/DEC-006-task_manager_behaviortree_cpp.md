# DEC-006：任务状态机采用 BehaviorTree.CPP，而不是手写 FSM

- 状态：确认
- 阶段：P1
- 日期：2026/4/17

## 背景

项目0的软件架构将任务决策层对应到 p0_semantic_nav 和 p0_task_manager，其中 p0_semantic_nav 负责语义导航，p0_task_manager 负责任务管理与任务组织。

迭代一到迭代二技术升级路径明确：任务状态机采用 BehaviorTree.CPP，迭代二通过扩展 BT 节点和 XML 增加复杂任务逻辑，不需要推翻重来。

项目0最终需要能够在答辩中说明“为什么做、怎么做、做成了什么、为什么有价值、后续还能怎么做”，任务组织方式会影响项目从普通导航小车向具身智能平台表达的完整性。

## 决策

项目0任务状态机采用：

> BehaviorTree.CPP。

具体约定：

1. 迭代一使用 BehaviorTree.CPP 组织基础任务流程。
2. 不采用临时手写 FSM 作为长期主路径。
3. p0_task_manager 负责围绕 BT 组织任务执行、任务状态、任务中止和结果反馈。
4. 迭代二通过增加 BT 节点和 XML 结构扩展更复杂任务逻辑。

## 理由

1. 迭代一到迭代二技术升级路径已明确任务状态机采用 BehaviorTree.CPP，并通过扩展 BT 节点和 XML 增强复杂任务逻辑。
2. BT 更适合表达任务执行、失败处理、恢复、中止和条件判断等结构化任务逻辑。
3. 如果迭代一先使用手写 FSM，迭代二很可能需要重构任务组织层，不符合“不推翻重来”的总路线。
4. BT 方案有利于后续成果表达，能更清楚展示系统如何从单点导航扩展到任务级行为组织。

## 影响

- 影响阶段：P5、P6、P9。
- 影响模块：p0_task_manager、p0_semantic_nav、p0_navigation、p0_system_manager。
- 影响文件：BT XML、任务节点实现、任务状态接口。
- 影响验收：任务启动、执行、失败、中止、恢复和结果反馈。

## 回退条件（可选）

如果迭代一中 BT 集成成本高于预期，可先实现极简 BT 树和少量节点，不扩展复杂任务逻辑；不建议退回临时 FSM，除非 BT 框架在环境或依赖上出现不可解决问题。
