# 项目0流程（第二版）

---

## 总体说明

### 推进原则
- 先做成，再做强，再做深
- 迭代一先独立完成并验收，再进入迭代二
- 单人solo开发，每阶段可执行、可验收、可回退
- 真机主线，仿真辅助，真机验收
- 成果表达与素材采集贯穿全程，不堆积到最后

### 阶段总览

| 阶段 | 名称 | 归属 |
|------|------|------|
| P0 | 项目定义收束 | 前置准备 |
| P1 | 功能落地、技术选型与工程准备 | 前置准备 |
| P2 | 底盘与基础运动成立 | 迭代一 |
| P3 | 感知与定位建图成立 | 迭代一 |
| P4 | 导航与避障成立 | 迭代一 |
| P5 | 语义导航与任务调度成立 | 迭代一 |
| P6 | 系统集成与迭代一验收 | 迭代一 |
| P7 | 迭代二增强方案设计 | 迭代二 |
| P8 | 迭代二前段：定位建图鲁棒性增强 | 迭代二 |
| P9 | 迭代二后段：语义导航能力深化 | 迭代二 |
| P10 | 实验收束与成果表达 | 收束 |

### 贯穿线（非独立阶段，伴随全程）
- **数据记录与日志**：从P2开始，每次真机运行必须记录bag文件、运行日志
- **Git版本管理**：从P1开始，所有代码、配置、文档纳入版本控制
- **过程文档与决策记录**：从P0开始，关键决策、问题、方案变更必须留痕
- **展示素材采集**：从P2开始，每个阶段保留运行视频、截图、数据可视化
- **学习笔记**：从P1开始，关键技术学习过程同步记录

## P0：项目定义收束

### 本质
将已完成的成熟定义文件确认锁定为开发依据，形成项目推进的起点基线。

### 主要任务
1. 确认锁定六份成熟文件：《最终系统定位》《客观限制》《主观需求》《迭代一核心能力》《迭代二增强能力》《整体成果形态》
2. 基于本流程文档形成粗粒度版本线路图（V0.1→V1.0→V2.0分别对应哪些阶段）
3. 形成粗粒度里程碑时间表（迭代一完成时间、迭代二完成时间、最终收束时间）
4. 确认校园网可用性及对开发部署的影响
5. 确认实验场地可用时间段与进场条件

### 输出成果
- 锁定版项目定义文件集
- 版本线路图
- 粗粒度里程碑时间表

### 依赖关系
- 前置依赖：无
- 后续阶段P1-P10均依赖此阶段输出

## P1：功能落地、技术选型与工程准备

### 本质
将迭代一十项核心能力展开为具体功能条目，确定每项功能的技术实现方案，完成硬件方案、系统架构设计和工程环境搭建，使项目具备开始动手的全部前提条件。

### 主要任务

#### 1.1 功能清单与技术选型
1. 将迭代一十项核心能力逐项展开为中粒度功能条目
2. 对每个功能条目区分：迭代一必须实现 / 迭代一接口预留 / 不纳入项目0
3. 对迭代一必须实现的功能条目，逐项确定技术选型与实现方案，包括但不限于：
   - 底层通信方案（串口/CAN/协议格式）
   - 电机控制方案（PID参数策略、编码器处理）
   - SLAM方案选型（如FAST-LIO2/LIO-SAM/Cartographer等，含选型理由）
   - 导航框架选型（如Nav2，含关键插件选择）
   - 语义导航实现方案（语义目标定义方式、目标转换机制、LLM/规则/混合）
   - 任务状态机方案（行为树/有限状态机/其他）
   - 安全机制方案（急停实现方式、禁区实现方式、异常检测方式）
   - 数据记录方案（rosbag2/自定义日志/数据库）
   - 系统管理方案（启动管理、状态监控、参数配置）
4. 输出《迭代一功能清单与技术选型表》

#### 1.2 系统架构设计
1. 系统总架构设计（模块划分、模块间关系、主链路与辅助链路）
2. 计算与通信架构（Orin NX与STM32职责划分、通信链路、ROS2节点规划）
3. 电源架构（电池→各设备供电路径、电压转换方案、电源保护）
4. 硬件架构（传感器安装位置、底盘结构布局、接线方案）
5. 软件架构（ROS2 workspace结构、包划分、话题/服务/动作接口初步定义、数据流与控制流）
6. 接口定义图(模块间话题/服务/动作接口定义、自定义消息结构与调用关系)
7. 输出各架构图与架构说明文档

#### 1.3 硬件方案与采购
1. 已有硬件盘点与状态确认
2. 缺失硬件确定（底盘结构件、电机驱动板、电压转换模块、补光模块、接线材料、紧急停止开关等）
3. BOM与采购清单输出
4. 启动采购

#### 1.4 工程环境搭建
1. Orin NX系统环境部署（Ubuntu、ROS2、驱动、开发工具链）
2. STM32开发环境搭建（工具链、烧录、调试）
3. Git仓库建立与文件夹结构建立
4. 仿真基础环境搭建（Gazebo或等效工具、走廊场景简模、小车URDF模型）
5. 远程开发与调试方案确认（SSH/远程桌面/VSCode Remote等）

#### 1.5 WBS与Sprint规划
1. 基于功能清单和技术选型，制定迭代一WBS（工作分解结构）
2. 将WBS按P2-P6阶段分配，形成各阶段Sprint任务清单
3. 每个Sprint明确验收标准

### 输出成果
- 《迭代一功能清单与技术选型表》
- 系统架构文档（含计算通信架构图、电源架构图、硬件架构图、软件架构图）
- 硬件BOM与采购清单
- 开发环境就绪确认记录
- 仿真基础环境就绪确认
- 迭代一WBS与Sprint规划
- Git仓库初始化完成

### 依赖关系
- 前置依赖：P0
- P2依赖：硬件到位、开发环境就绪、底层通信方案与电机控制方案已确定
- 采购周期可能与P1其他子任务并行，硬件未到位期间优先推进软件环境和仿真环境

## P2：底盘与基础运动成立

### 本质
让小车在物理世界中能动、能停、能被人工接管，建立一切后续能力的物理前提。

### 主要任务
1. 底盘机械组装（结构件、电机安装、轮子、传感器支架）
2. 电源系统组装与测试（电池→电压转换→各设备供电验证）
3. STM32底层固件开发：
   - 电机驱动与PWM控制
   - 编码器读取与速度计算
   - PID速度闭环控制
   - 与Orin NX的通信协议实现（串口或CAN）
   - 底层安全：超时停车、通信丢失停车
4. Orin NX端底层驱动节点：
   - 速度指令下发
   - 里程计数据接收与发布（odom话题、TF变换）
   - 底层状态接收（电压、电流、异常标志）
5. 遥控接管功能实现（手柄或键盘 → cmd_vel → 底层执行）
6. 物理急停开关安装与测试
7. 真实场地基础运动验证：前进、后退、转向、停止、速度响应
8. 可选：仿真中预验证运动控制节点逻辑

### 输出成果
- 可遥控运动的实体小车
- 底层通信协议文档
- 里程计数据发布正常
- 急停与人工接管功能可用
- 电源系统工作正常
- 底层固件源码与说明
- 第一段真机运行视频素材

### 验收标准
- 小车可通过遥控在走廊中稳定行驶，速度可控
- 里程计数据与实际运动基本一致
- 急停按下后小车立即停止
- 通信中断后小车自动停止

### 依赖关系
- 前置依赖：P1（硬件到位、环境就绪、方案已定）
- P3依赖：底盘能动、里程计可用、遥控可用（用于带着小车采集传感器数据）

## P3：感知与定位建图成立

### 本质
让小车能看见环境、知道自己在哪、能构建环境地图，建立导航的空间基础。

### 主要任务
1. Mid360激光雷达驱动上线与点云发布验证
2. D435深度相机驱动上线与RGB/深度图发布验证
3. IMU数据发布与校准（Mid360自带IMU）
4. 补光模块安装与控制（应对夜间低光照）
5. 传感器外参标定（雷达-IMU、相机-雷达、相机-车体）
6. SLAM链路搭建（按P1选型方案实施）
7. 真实走廊场景首次建图：
   - 遥控带车走完回字形走廊
   - 检查点云质量、地图完整性、回环效果
8. 基础定位验证：
   - 在已有地图上重定位
   - 回字形走一圈检查漂移程度
9. 高风险区域在地图中标注（楼梯口、天井边缘、电梯口、排水槽）
10. 地面风险感知初步验证（排水槽、落差区域是否可被感知）
11. 可选：仿真中预跑SLAM链路，降低真机调试成本

### 输出成果
- 各传感器数据正常发布与可视化确认
- 传感器标定参数
- 走廊环境基础地图（保存为可复用格式）
- 实时定位链路可用（位姿输出稳定）
- 高风险区域标注完成
- 建图过程bag文件（用于后续迭代二对比实验基线）
- 补光模块工作正常

### 验收标准
- 回字形走廊建图完整，无明显断裂或严重畸变
- 重定位后走一圈，终点与起点偏差在可接受范围内
- 高风险区域在地图中有明确标识

### 依赖关系
- 前置依赖：P2（底盘运动、里程计、遥控可用）
- P4依赖：定位可用、地图可用、高风险区域已标注

## P4：导航与避障成立

### 本质
让小车能自主从A走到B，能避开障碍物，能遵守高风险区域约束，形成自主导航闭环。

### 主要任务
1. 导航框架部署与配置（按P1选型方案实施）
2. 代价地图配置：
   - 静态层（基础地图）
   - 障碍物层（雷达实时检测）
   - 高风险禁区层（楼梯口、天井边缘等，设为禁止通行或高代价区域）
   - 排水槽/落差区域代价配置
3. 全局路径规划调试（走廊环境下路径合理性）
4. 局部避障调试（动态障碍物绕行、窄通道通过）
5. 路径跟踪控制器调试（速度平滑、转弯行为）
6. 目标点导航闭环测试：指定目标→规划→执行→到达判定
7. 多目标点连续导航测试（走廊内多个位置依次到达）
8. 导航异常处理：
   - 路径被阻断时重规划
   - 长时间卡死检测与安全停车
   - 接近高风险区域时减速或停止
9. 自设障碍物场景测试

### 输出成果
- 自主导航闭环可用
- 高风险区域约束生效
- 基础避障可用
- 导航异常处理基础可用
- 导航过程数据记录（路径、速度、事件）

### 验收标准
- 可自主导航到走廊内任意指定目标点
- 遇到障碍物能绕行或安全停车
- 不会规划出经过楼梯口/天井边缘的路径
- 连续导航多个目标点无严重故障

### 依赖关系
- 前置依赖：P3（定位、地图、高风险区域标注）
- P5依赖：导航闭环可用

## P5：语义导航与任务调度成立

### 本质
让小车能理解"去哪里"的语义指令并执行，具备基础任务管理能力，使系统从"导航工具"升级为"具身智能体雏形"。

### 主要任务
1. 语义目标定义机制建立：
   - 定义场景中的语义位置/区域（如"厕所门口""楼梯口A""电梯口"等）
   - 建立语义标签与地图坐标/区域的映射关系（语义位置表或语义地图初步结构）
2. 语义目标→导航目标转换链路实现：
   - 接收语义指令（文本/指定接口）
   - 解析语义目标
   - 查找对应导航目标
   - 下发给导航模块执行
3. 基础任务状态机实现：
   - 任务接收→任务执行→执行监控→任务完成/任务失败
   - 任务中止与恢复
   - 多任务排队或拒绝策略
4. 行为模式管理：
   - 自主模式 / 手动遥控模式 / 急停模式
   - 模式切换逻辑与安全约束（急停模式优先级最高）
5. 语义导航端到端测试：
   - 语义指令→解析→导航→到达→结果反馈
   - 无效语义指令处理
   - 导航失败时的任务级反馈
6. 可选：简单语音输入接口或LLM文本解析接口接入

### 输出成果
- 语义位置表/语义地图初步结构
- 语义导航基础链路可用
- 任务状态机可用
- 模式管理可用
- 语义导航端到端测试记录

### 验收标准
- 输入语义指令（如"去厕所"），小车能正确导航到对应位置
- 输入无效指令时系统能正确拒绝或反馈
- 模式切换正常，急停模式可随时介入
- 任务执行过程中状态可观测

### 依赖关系
- 前置依赖：P4（导航闭环可用）
- P6依赖：语义导航与任务调度可用

## P6：系统集成与迭代一验收

### 本质
将P2-P5各阶段成果集成为完整系统，补全系统级管理能力，进行全链路联调与验收测试，确认迭代一平台真正成立。

### 主要任务

#### 6.1 系统级能力补全
1. 系统启动与关闭流程规范化（一键启动脚本、启动顺序管理、优雅关闭）
2. 系统自检流程（传感器在线检查、底层通信检查、定位状态检查、电量检查）
3. 状态监控与异常分级（正常/警告/错误/致命，各级对应处理策略）
4. 参数配置管理（关键参数可配置、不硬编码）
5. 电源监控与低电量保护
6. 运行状态可视化（RViz/自定义面板/终端状态输出）

#### 6.2 全链路联调
1. 完整流程测试：启动→自检→建图或加载地图→定位→接收语义任务→导航执行→避障→安全约束→任务完成→数据记录→关闭
2. 多轮连续运行测试
3. 异常场景联调：通信中断、传感器故障、导航失败、低电量、人工接管介入

#### 6.3 迭代一验收测试
1. 对照《迭代一阶段要具备的核心能力》十项逐项验收
2. 安全专项测试：
   - 高风险区域约束有效性
   - 急停响应时间
   - 异常停车可靠性
   - 人工接管切换可靠性
3. 基础性能记录：定位精度初步评估、导航成功率、语义导航成功率、单次运行时长
4. 已知问题与局限性记录

#### 6.4 迭代一阶段成果输出
1. 迭代一平台报告撰写（覆盖：约束、目标、功能框架、技术选型、架构、各模块实现、联调测试、安全机制、运行表现、问题与后续方向）
2. 迭代一演示视频录制
3. 迭代一版本代码与配置归档（Git标签：V1.0）
4. 迭代二输入整理：问题清单、性能基线数据、待增强方向确认

### 输出成果
- 系统级管理能力可用
- 全链路联调通过
- 迭代一十项核心能力验收记录
- 安全专项测试记录
- 性能基线数据（作为迭代二对比基准）
- 迭代一平台报告
- 迭代一演示视频
- V1.0版本归档
- 迭代二问题清单与增强方向确认

### 验收标准
- 十项核心能力全部达到"基础成立"标准
- 全链路可完整运行，无致命故障
- 安全机制全部生效
- 平台报告和演示视频完成

### 依赖关系
- 前置依赖：P2-P5全部完成
- P7依赖：迭代一验收通过、性能基线数据可用、问题清单已整理

### 里程碑
**迭代一完成，目标时间：大三上结束之前**

## P7：迭代二增强方案设计

### 本质
基于迭代一真实运行暴露的问题和性能基线，设计迭代二的增强方案与实验方案，而不是在平台未成立时空想。

### 主要任务
1. 分析迭代一定位建图在真实场景中的具体问题（长走廊漂移、回环失败、退化场景、长时运行稳定性等）
2. 确定迭代二前段增强的具体技术路线：
   - 多传感器融合增强方案（雷达+IMU+编码器+视觉的融合策略与方法选型）
   - 回环检测增强方案
   - 退化场景处理方案
   - 其他鲁棒性增强手段
3. 设计迭代二前段实验方案：
   - 基线方案定义（迭代一SLAM配置）
   - 增强方案定义
