## 二、选型总原则

1. **避免推翻重来**：迭代一选定的技术栈，迭代二必须能在其基础上增强、扩展或替换插件，而不是整体推翻重构 
2. **真机主线**：所有选型必须在Orin NX + STM32F407真机上可运行 
3. **单人可落地**：优先选择社区成熟、文档充足、学习成本可控的方案 
4. **与已有硬件匹配**：充分利用Orin NX 16GB、Mid360、D435、STM32F407，不为选型反向更换硬件
5. **迭代二可承接**：选型必须为迭代二的多传感器融合增强、回环检测增强、语义导航深化预留清晰的升级路径

## 四、核心选型详细说明

---

### ① 底层通信方案（Orin NX ↔ STM32F407）

**选型结论：UART串口 + 自定义二进制协议帧**

#### 迭代边界说明

- **迭代一**：上位机（Orin NX）与下位机（STM32F407）之间通过UART串口通信。上位机下发速度指令（线速度、角速度），下位机上传编码器数据和IMU数据。通信协议为自定义二进制帧格式（帧头 + 数据 + 校验 + 帧尾）
- **迭代二**：通信协议本身不需要推翻。如需增加传输数据种类（如额外传感器数据），只需扩展协议帧的数据字段。如未来需多设备挂载，可在架构上迁移到CAN总线，但迭代一不需要为此做额外开发

#### 选型理由

1. 迭代一只有一个下位机，通信需求简单（速度指令 + 编码器/IMU回传），UART完全满足
2. STM32F407和Orin NX均原生支持UART，无需额外硬件
3. 自定义协议帧实现简单（百行级代码），调试工具丰富（串口助手可直接查看原始数据）
4. 单人学习成本最低，出问题最容易排查
5. 迭代二不需要更换通信方案，只需扩展数据字段

#### 已知风险

- UART波特率有上限（通常115200-921600），但迭代一数据量远未触及上限
- 无硬件级仲裁，多设备扩展需额外串口或迁移到CAN——但迭代一只有一个下位机，不存在此问题

#### 关键实现要点

- 通信频率：建议50-100Hz
- 校验方式：CRC8或CRC16
- 异常处理：超时检测（下位机超过200ms未收到指令则自动停车）、校验失败丢帧

### ② 电机控制方案（快速确认）

**选型结论：增量式PID速度闭环控制，运行在STM32F407上**

#### 迭代边界说明

- **迭代一**：STM32F407读取编码器脉冲，计算实际轮速，通过PID控制器调节PWM输出，实现左右电机速度闭环。里程计由编码器数据在STM32上初步计算后上传，或在Orin NX侧基于编码器原始数据计算
- **迭代二**：电机控制层不需要变动。迭代二的多传感器融合增强发生在上位机SLAM/定位层面（将编码器里程计作为额外约束源融入定位系统），不改变底层电机控制逻辑

#### 选型理由

1. 增量式PID是电机速度控制的标准方案，STM32教程极多
2. STM32F407性能充裕，可同时处理双电机PID + 编码器读取 + IMU读取 + 串口通信
3. 迭代二不需要修改此层

#### 关键实现要点

- PID控制周期：建议10-20ms
- 编码器：AB相正交解码，STM32定时器硬件解码
- PWM频率：建议10-20kHz

### ③ SLAM方案选型（建图 + 定位）

**选型结论：FAST-LIO2**

#### 迭代边界说明（重要，请仔细阅读）

- **迭代一中FAST-LIO2的角色**：FAST-LIO2是一个LiDAR+IMU紧耦合的里程计/建图系统。在迭代一中，它使用的传感器**仅为Mid360激光雷达和其内置IMU（或外置IMU）**。它**不融合轮式编码器里程计**，也**不融合D435深度相机**。迭代一的FAST-LIO2就是一个纯LiDAR-Inertial系统
- **迭代一中编码器里程计的角色**：编码器里程计在迭代一中由底盘层独立计算并发布，供Nav2导航框架作为基础运动反馈使用，但**不参与FAST-LIO2的定位融合**
- **迭代一中D435的角色**：D435在迭代一中仅作为深度/视觉数据源接入系统，用于辅助感知和调试观测，但**不参与FAST-LIO2的定位融合**
- **迭代二的增强空间**：正因为迭代一只用了LiDAR+IMU两个信息源，迭代二才有清晰的增强路径 
  - 增强1：将轮式编码器里程计融入定位系统（多传感器融合）
  - 增强2：将D435视觉/深度信息融入定位系统（视觉辅助）
  - 增强3：外挂回环检测模块（如Scan Context + 位姿图优化），解决130m回字形走廊的累积漂移
  - 增强4：退化场景检测与处理（长直走廊几何退化时切换/融合其他信息源）
  - 每项增强都可以独立做消融实验，量化其贡献
- **为什么不需要推翻重来**：迭代二的增强是在FAST-LIO2输出的基础上"往上叠加"，而不是替换FAST-LIO2本身。FAST-LIO2作为前端里程计继续运行，增强模块在其输出之上工作

#### 选型理由

1. **与Mid360高度适配**：FAST-LIO2由港大MARS实验室开发，Livox官方生态直接推荐，对非重复扫描雷达的支持是原生的
2. **计算效率极高**：基于迭代扩展卡尔曼滤波（IEKF），Orin NX运行毫无压力，留出充裕算力给导航和其他模块
3. **建图质量好**：在130m回字形走廊尺度下，FAST-LIO2的里程计精度足够完成迭代一建图和定位
4. **ROS2社区有成熟移植版**：不需要自己从ROS1移植
5. **迭代二增强路径清晰**：如上所述，四个增强方向层次分明，每个都可独立验证
6. **单人学习成本低**：配置简单，社区资料多，调参文档充足
7. **不存在"好高骛远"问题**：使用3D雷达的开源SLAM系统不等于"自研3D SLAM"，使用难度与2D SLAM同级

#### 已知风险

- FAST-LIO2无回环检测，130m回字形走廊存在累积漂移——迭代一可接受（建一次图、多次定位运行，漂移在可控范围内），迭代二外挂回环模块解决
- 长直走廊可能出现几何退化（LiDAR特征不足）——迭代一通过IMU辅助缓解，迭代二通过编码器/视觉融合进一步增强

#### 备选方案

- LIO-SAM：自带回环检测，但适配Mid360需额外工作，且回环能力放在迭代一会压缩迭代二研究空间
- Cartographer 3D：ROS2支持好，但对非旋转雷达适配一般，长走廊退化处理不如LIO系列

### ④ 导航框架与插件选型

**选型结论：Nav2框架 + NavFn全局规划器 + RPP局部控制器**

#### 迭代边界说明

- **迭代一**：使用Nav2标准框架，NavFn做全局路径规划，RPP（Regulated Pure Pursuit）做局部路径跟踪与控制。代价地图使用Nav2默认的静态层+障碍物层+膨胀层配置。高风险区域（楼梯口、天井边缘等）通过keepout_filter（禁区层）标注在地图中 
- **迭代二**：Nav2框架不变，插件可按需替换或增加。例如局部控制器可升级为MPPI（模型预测路径积分控制器），代价地图可增加自定义层。**框架级别无需推翻重来**，只需替换插件或调整配置

#### 选型理由

1. **Nav2是ROS2生态的标准导航框架**，社区最成熟，文档最全，无其他合理替代
2. **NavFn**：走廊环境以直道和直角拐弯为主，Dijkstra/A*规划完全够用，配置最简单
3. **RPP（Regulated Pure Pursuit）**：路径跟踪平滑，内置速度调节（接近障碍物/拐弯时自动减速），非常适合走廊场景。比DWB在直道跟踪上更稳，比MPPI配置简单得多
4. **禁区层（keepout_filter）**：Nav2原生支持，可直接在地图上标注楼梯口等高风险区域为禁行区，导航时自动回避

#### 已知风险

- RPP的动态避障能力弱于DWB——走廊场景动态障碍物少，可接受；若实测不足，可切换为DWB（Nav2插件化设计，切换只需改配置文件）
- Nav2参数较多，初次调参有学习曲线——但社区有大量走廊/室内场景的参数模板可参考

#### 备选方案

- 局部控制器备选：DWB（如RPP动态避障不足时切换）
- 全局规划器备选：Smac Planner 2D（如需更平滑路径）

### ⑤ 语义导航实现方案

**选型结论：YAML语义位置表 + 规则匹配**

#### 迭代边界说明

- **迭代一**：预定义一张YAML格式的语义位置表，将语义标签（如"厕所""楼梯口A""电梯口"）映射到地图坐标。系统接收语义指令后，通过精确字符串匹配在表中查找对应坐标，转换为Nav2导航目标点。整个链路是确定性的、完全透明的、无ML推理的
- **迭代二后段**：语义导航深化。可能的增强方向包括：
  - 语义目标与空间区域的更丰富关联
  - 引入本地小模型LLM做自然语言→结构化指令解析
  - 语义地图初步能力
  - **YAML语义表本身作为基础数据结构保留**，LLM增强是在其之上叠加，不是推翻

#### 选型理由

1. 迭代一语义导航的目标是"链路成立"而非"语义智能" 
2. YAML表 + 规则匹配可在1-2天内完成开发和验证
3. 链路完全确定性，调试透明，不存在ML推理不确定性
4. 语义位置表本身就是迭代二语义地图增强的基础数据结构，不会被废弃
5. 迭代二引入LLM后，可形成"迭代一规则基线 vs 迭代二LLM增强"的对比实验

#### 已知风险

- 只能精确匹配预定义标签，无自然语言理解能力——迭代一可接受，迭代二解决

### ⑥ 任务状态机方案

**选型结论：BehaviorTree.CPP（Nav2内置行为树框架）**

#### 迭代边界说明

- **迭代一**：使用Nav2自带的BehaviorTree.CPP框架，在XML中定义任务行为树。迭代一的任务逻辑较简单（接收任务→调用导航→监控状态→处理异常→反馈结果），可在Nav2默认BT XML基础上扩展
- **迭代二**：任务逻辑变复杂时（多步骤任务、条件分支、多传感器触发等），行为树天然支持扩展，只需增加新节点和修改XML，**不需要推翻框架**。如果迭代一选了手写FSM，迭代二大概率需要重构为BT，违反"避免推翻重来"原则

#### 选型理由

1. **Nav2本身就用BehaviorTree.CPP**管理导航行为，你无论如何都要接触它——不如直接用它做任务调度，避免两套框架并存
2. 行为树的扩展性远强于FSM，迭代二可持续增强
3. BT的XML定义方式结构清晰，可视化工具（Groot）支持调试
4. 学习成本虽比手写FSM高，但Nav2文档中BT部分教程充足，且这个投入在迭代二会持续回报
5. **关键：选BT避免了迭代二推翻重来的风险**

#### 已知风险

- BT学习曲线比手写FSM陡——建议在P5阶段（语义导航与任务调度成立）时集中学习，前面P2-P4阶段先用Nav2默认BT不做大改

#### 备选方案

- 手写FSM：如果BT学习成本确实超出预期，可退回FSM作为迭代一方案，但需接受迭代二重构风险

### ⑦ 安全机制方案

**选型结论：硬件急停按钮 + Nav2禁区层 + 软件级安全节点**

#### 方案说明

| 安全能力 | 实现方式 | 说明 |
|----------|---------|------|
| 硬件急停 | 物理急停按钮直接切断电机驱动供电 | 不依赖任何软件，最高优先级安全保障 |
| 高风险区域禁区 | Nav2 keepout_filter | 在地图上标注楼梯口、天井边缘等为禁行区域，导航时自动回避  |
| 软件急停 | ROS2安全节点发布急停话题 | 收到急停消息后停止所有运动指令 |
| 异常停车 | 安全节点监控通信/定位/感知状态 | 检测到异常时自动发布停车指令 |
| 速度限幅 | 底层STM32 + 上层Nav2参数 | 双层限速保护 |
| 人工接管 | 遥控器信号优先级高于自主控制 | 随时可切换为人工遥控 |

#### 迭代边界说明

- **迭代一**：上述全部实现。安全是底线能力，不分迭代
- **迭代二**：安全机制不需要变动，只需根据新增能力扩展监控项

### ⑧ 数据记录方案

**选型结论：rosbag2 + ROS2标准日志（rclcpp/rclpy logger）**

#### 方案说明

| 记录需求 | 实现方式 | 说明 |
|----------|---------|------|
| 传感器原始数据 | rosbag2录制关键话题 | 激光雷达、IMU、深度相机、里程计等话题 |
| 过程日志 | ROS2标准logger | 各节点使用RCLCPP_INFO/WARN/ERROR输出日志 |
| 结果数据 | rosbag2 + 日志文件 | 导航路径、任务状态、安全事件等 |
| 调试支撑 | RViz2 + rosbag2回放 | 事后可回放bag文件在RViz2中复现运行过程 |

#### 迭代边界说明

- **迭代一**：rosbag2录制 + 标准日志，足够支撑调试和复盘
- **迭代二**：rosbag2录制的bag文件直接作为实验基线数据 [5]，可复用于对比实验。不需要换方案

#### 选型理由

1. rosbag2是ROS2标准数据记录工具，零额外开发
2. bag文件可离线回放，支持事后分析和实验复现
3. 迭代二的对比实验/消融实验直接复用迭代一bag文件作为基线

### ⑨ 系统管理方案

**选型结论：ROS2 Launch文件 + 自定义系统管理节点 + Shell脚本**

#### 方案说明

| 管理需求 | 实现方式 | 说明 |
|----------|---------|------|
| 启动管理 | ROS2 Launch文件（Python格式） | 定义启动顺序、参数加载、节点组织 |
| 一键启动 | Shell脚本封装Launch | 不同模式（建图/定位运行/任务模式）各一个脚本 |
| 模式管理 | 自定义系统管理节点 | 管理当前系统模式（手动/自主/急停/建图/定位运行），控制模式切换规则 |
| 状态监控 | 自定义系统管理节点 | 监控关键模块在线/离线状态，输出系统整体状态 |
| 自检 | 启动时由系统管理节点执行 | 检查传感器、通信、定位条件，输出自检结果 |
| 参数配置 | ROS2参数文件（YAML） | 关键参数可配置，不硬编码 |

#### 迭代边界说明

- **迭代一**：上述全部实现，但功能保持简洁（自检项有限，模式管理逻辑简单）
- **迭代二**：在同一节点上扩展监控项和自检项，不需要重构

## 七、迭代一→迭代二技术升级路径总览

为避免误解，以下汇总迭代一和迭代二在技术栈上的关系：

| 技术域 | 迭代一方案 | 迭代二变化 | 是否需要推翻重来 |
|--------|-----------|-----------|-----------------|
| 底层通信 | UART + 自定义协议帧 | 不变（如需扩展数据则扩展帧字段） | ❌ 不需要 |
| 电机控制 | STM32增量式PID | 不变 | ❌ 不需要 |
| SLAM | FAST-LIO2（LiDAR+IMU） | 在FAST-LIO2基础上叠加：+编码器融合/+视觉融合/+回环/+退化处理 | ❌ 不需要（FAST-LIO2继续作为前端） |
| 导航框架 | Nav2 + NavFn + RPP | Nav2不变，可替换插件（如RPP→MPPI） | ❌ 不需要 |
| 语义导航 | YAML表 + 规则匹配 | 在YAML表基础上叠加LLM增强/语义地图 | ❌ 不需要（YAML表保留为基础结构） |
| 任务状态机 | BehaviorTree.CPP | 扩展BT节点和XML，增加复杂任务逻辑 | ❌ 不需要 |
| 安全机制 | 硬件急停+Nav2禁区+安全节点 | 扩展监控项 | ❌ 不需要 |
| 数据记录 | rosbag2 + ROS2 logger | 不变（迭代一bag可直接作为基线数据） | ❌ 不需要 |
| 系统管理 | Launch + 系统管理节点 | 扩展监控项和自检项 | ❌ 不需要 |

**结论：所有技术选型在迭代二均为增量式增强，不存在推翻重来。**
