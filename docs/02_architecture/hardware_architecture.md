# 1. 文档定位

本文档属于项目0在 P1 阶段的系统架构设计输出物之一，放置在 `docs/02_architecture/` 目录下，与 `system_architecture.md`、`compute_comm_architecture.md`、`power_architecture.md`、`software_architecture.md` 并列管理。
根据项目文件结构设计，`hardware_architecture.md` 是 P1 阶段正式架构文档之一，用于承接系统总架构中未展开的硬件实现层内容。

本文档的目标不是直接替代实物装配图，也不是一次性把所有机械和电气细节做死，而是为迭代一平台提供一个**真实可运行、可维护、可扩展、可联调、可验收**的硬件基线方案，以支撑后续 P2-P6 的机械组装、传感器接入、底盘调试、导航联调和系统集成验收。

# 2. 阶段归属与依赖关系

项目0总体流程采用 P0-P10 分阶段推进方式，其中 P1 的本质是“功能落地、技术选型与工程准备”，P2 才进入“底盘与基础运动成立”，P3 进入“感知与定位建图成立”，P4 进入“导航与避障成立”，P5 进入“语义导航与任务调度成立”，P6 进入“系统集成与迭代一验收”。

因此，硬件架构文档属于 P1 的正式输出，直接服务于以下后续阶段：

- **P2**：底盘机械组装、急停安装、人工接管链路建立、基础运动验证；
- **P3**：Mid360 与 D435 的实际安装、接入、标定、建图与定位验证；
- **P4-P6**：导航、安全、系统管理、运行监控、自检与集成验收；

项目推进强调“先做成，再做强，再做深”“单人 solo 开发，每阶段可执行、可验收、可回退”“真机主线，仿真辅助，真机验收”，因此本硬件架构以**迭代一平台成立优先**为首要原则，而非追求高复杂度机械设计或一次性把所有扩展件装满。

# 3. 架构目标

本硬件架构面向教学楼室内走廊场景的地面移动具身智能平台，目标是为系统提供稳定的物理载体，使平台具备感知—定位—导航—语义理解—安全保护的闭环运行基础。

具体目标如下：

1. 支撑四轮差速底盘在真实场景中稳定运动、停止和人工接管；
2. 支撑 Mid360 与 D435 的稳定安装、接入和后续标定；
3. 与已确定的计算与通信架构一致，即 Orin NX 作为主计算平台，STM32 作为底层执行控制平台；
4. 与已确定的电源架构一致，即采用单主电池统一供能，划分动力支路、计算支路、传感器/控制支路，并保留硬件急停只切动力支路的安全边界；
5. 兼顾调试便利性、安全性、维护性、可展示性与迭代扩展接口预留；

# 4. 上位架构约束

## 4.1 计算与通信约束

根据已确定的计算与通信架构，系统主要硬件实体及连接关系如下：

- 外部操作者
  - 硬件急停按钮 → 纯硬件电路 → 切断电机驱动供电；
- Orin NX（主计算平台）
  - USB / 以太网 ← Mid360（点云 + IMU）；
  - USB ← D435（RGB + 深度）；
  - UART ←→ STM32（速度指令下发 / 编码器与底层状态上传）；
- STM32（底层控制平台）
  - 定时器 ← 编码器；
  - PWM → 电机驱动板；
  - UART ←→ Orin NX；

此外，软件架构明确：ROS2 主业务统一运行在 Orin NX 上，STM32 不运行 ROS2 主业务，仅作为独立固件平台通过底盘桥接接入 ROS2 世界。
这意味着硬件布局必须服从“**主计算在 Orin、实时执行在 STM32、感知接入在 Orin、底盘闭环在 STM32**”这一基本边界。

## 4.2 电源架构约束

已定电源架构的核心约束包括：

1. 采用单主电池统一供能；
2. 电源分为动力支路、计算支路、传感器/控制支路三大部分；
3. 动力支路由主电池直供驱动板，急停只切动力支路；
4. Orin NX 独立高优先级稳压供电，不受急停切断；
5. STM32、Mid360、D435、补光控制挂载于独立传感器/控制支路；
6. 迭代一正式实现电池电压监测、低电量告警和低电量保护基础能力；
7. 补光模块纳入系统控制，作为弱光视觉支撑链路的一部分。

因此，硬件架构必须为电池、总电源入口、驱动板、Orin NX 稳压支路、传感器/控制支路、急停按钮和补光模块预留明确物理位置与走线路径。

# 5. 已确认的硬件决策

基于当前阶段已经确认的方案，项目0迭代一硬件架构采用以下硬件决策：

1. **底盘形式**：四轮底盘，采用左右两侧等效差速控制抽象，与 README 中的四轮差速底盘基线一致。
2. **车体结构**：现成底盘板 / 通用板材方案，双层板之间采用立柱连接，优先追求“快速装起来”。
3. **主计算平台**：Jetson Orin NX 16GB，已安装 SSD、外壳、小风扇散热与天线，作为 ROS2 主业务承载平台。
4. **底层控制平台**：STM32，作为底层执行与状态上传控制平台。
5. **主传感器**：Livox Mid360，顶部接近几何中心安装，优先保证视场，允许明显高出上层板。
6. **辅助视觉传感器**：Intel RealSense D435，前向安装，作为前向辅助视觉/深度感知、调试与记录数据源，不承担迭代一正式 FAST-LIO2 主融合输入。
7. **补光模块**：纳入系统控制，布置在 D435 附近，服务弱光视觉支撑。
8. **人工接管方案**：飞智黑武士2 + 2.4G USB 接收器接入 Orin NX，由上位机 teleop/手动控制节点生成 `cmd_vel`，再经底盘桥接下发至 STM32。
9. **急停方案**：急停按钮布置于车体顶部中后部，明显凸起，但要兼顾搬运时不易误触；其功能为通过纯硬件链路切断电机驱动供电。
10. **电机驱动方案**：两块双路驱动板，底层驱动四个电机，预留自然散热空间。
11. **电池固定方案**：主电池采用托盘式 + 绑带辅助固定。

# 6. 整车分层布局原则

双层底盘采用“**下层动力与供电、上层计算与控制、顶部感知支架**”的三层思路组织布局。
这种分层方式符合项目0在单人开发条件下对可维护性、可观察性和可调试性的要求，也与后续硬件照片采集、视频展示、模块替换和问题定位更匹配。

总体原则如下：

1. **下层优先放重、放高电流、放动力件**，降低重心并缩短动力走线；
2. **上层优先放计算与控制件**，方便调试、接线和维护；
3. **顶部优先放感知件**，尽量减少本体遮挡，稳定外参关系；
4. **动力线、计算线、传感器线分区走线**，避免后续维护混乱；
5. **优先保证安全和功能成立，再追求外观紧凑和高度压缩**，尤其在 Mid360 视场与整车紧凑性冲突时，优先 Mid360 视场。

# 7. 下层布局：动力与供电层

## 7.1 主电池

主电池是整车统一能源入口，应安装在**下层中部偏后**位置。
原因如下：

- 有利于压低整车重心，提高走廊运行稳定性；
- 有利于与动力支路、驱动板形成较短路径，减少高电流长走线；
- 为前部 D435 与补光模块、顶部 Mid360 支架留出更多安装空间；

电池固定方式采用**托盘式 + 绑带辅助固定**，即通过托盘限制电池前后左右滑动，通过绑带限制垂向跳动和振动脱落。
该方案在快速装配、可靠固定与后续维护之间取得平衡，适合迭代一阶段。

## 7.2 电机驱动板

两块双路电机驱动板安装在**下层靠近电池和电机线束出口的位置**。
推荐为：左右各一块，或中部并列但分开布置，以缩短驱动板到对应侧电机的动力线长度。

布局原则：

- 驱动板尽量靠近动力入口与电机出口；
- 两块驱动板之间不要紧贴，预留自然散热空间；
- 暂不额外增加主动散热，但保留后续加散热片或导风的余量；
- 不让高电流动力线穿越上层计算区。

## 7.3 下层电源相关器件预留

虽然更细的接线图和电气细化不在本版本展开，但下层必须预留以下器件安装区：

- 主电池输出接口；
- 主保护器件 / 总开关；
- 急停回路相关线路入口；
- 动力支路至驱动板分配路径；
- 计算支路和传感器/控制支路的 DC-DC 安装空间；

# 8. 上层布局：计算、控制与人工接管层

## 8.1 Orin NX

Orin NX 是系统唯一主计算与主控平台，负责感知接入、底盘桥接、定位、建图、地图管理、导航、语义导航、任务管理、安全管理、系统管理和数据记录等软件主业务与辅助业务。
因此，Orin NX 应安装在**上层中部偏前或中部偏中位置**，满足以下要求：

- 方便连接 Mid360、D435 的数据线；
- 保证风扇散热不被其他结构件完全遮挡；
- 方便插拔调试线、网线、串口线与供电线；

因为软件主业务集中在 Orin NX，上层布局应优先保证其可维护性，而不是将其藏入不易操作的位置。

## 8.2 STM32

STM32 是独立固件控制平台，负责编码器读取、电机控制、底盘保护、底层状态上传等实时闭环任务。

建议 STM32 安装在**上层靠后或靠侧后区域**，原因是：

- 靠近下层驱动板和底层执行链路，便于引出控制与状态线；
- 靠近 Orin NX，便于 UART 链路简洁；
- 便于烧录、调试、查看状态灯；
- 不与前向感知器件争抢空间。

## 8.3 手柄人工接管接入

迭代一人工接管方案采用**飞智黑武士2 + 2.4G USB 接收器接 Orin NX**。
该方案对应 P2 阶段“手柄或键盘 → `cmd_vel` → 底层执行”的人工控制路径，属于上位机 teleop 控制方案。

控制链路为：

```text
飞智黑武士2手柄
→ 2.4G USB接收器
→ Orin NX
→ teleop / 手动控制节点
→ cmd_vel
→ p0_base_bridge
→ UART
→ STM32
→ PWM
→ 电机驱动板
→ 电机
```

该方案的优点是快速落地、实现简单、符合迭代一平台先成立原则，但其安全等级低于“遥控接收机直连 STM32 的底层独立接管链路”。
因此，**硬件急停仍然是最高优先级安全手段**，不能被手柄人工控制替代。

硬件上应保证：

- Orin NX USB 口可稳定接入 2.4G 接收器；
- 接收器安装不应被金属外壳完全屏蔽；
- 接收器位置不应妨碍 Orin NX 其他接口维护。

# 9. 顶部布局：感知支架与传感器层

## 9.1 Mid360 安装方案

Mid360 是迭代一阶段的主空间感知传感器，其点云与内置 IMU 数据直接进入 Orin NX，并支撑后续 SLAM / 定位主路径。

安装原则如下：

1. **安装在车体顶部，尽量接近几何中心**；
2. **允许明显高出上层板**；
3. **优先保证视场，必要时牺牲整车紧凑性**；
4. 尽量避免被 Orin NX、上层板、电池壳体或其他突起件遮挡；
5. 支架结构应刚性足够，避免显著晃动影响点云稳定性。

之所以这样设计，是因为项目场景是教学楼室内走廊，平台后续要完成建图、定位、导航和高风险区域安全约束验证，而感知质量与遮挡情况会直接影响这些功能的成立。

## 9.2 D435 安装方案

D435 作为前向辅助视觉/深度传感器，建议安装在**车头上层前沿或前向支架区域**。
其职责是：

- 前向辅助感知；
- 调试可视化；
- 数据记录；
- 为弱光场景下的视觉支撑提供接口基础。

由于迭代一正式 IMU 主路径来自 Mid360 内置 IMU，D435 不承担迭代一 FAST-LIO2 正式融合主角色，因此其安装优先级低于 Mid360，但仍应保证：

- 视场前向无遮挡；
- 尽量减少与补光模块之间的遮挡或反光干扰；
- 线束可稳定固定，不因转弯或碰撞轻易松脱。

## 9.3 补光模块安装方案

补光模块纳入系统控制，是弱光视觉支撑链路的一部分。
在硬件布局上，补光模块建议与 D435 同属前向感知区域，但**不强行与 D435 做结构一体化**。
推荐布置方式为：

- 优先安装在 D435 上方偏前位置；
- 光线略向前下方覆盖；
- 不遮挡 D435 主视场；
- 尽量避免直射造成近距离地面过曝。

这样做的原因是：
迭代一重点是让弱光感知能力“先成立”，而不是在补光与相机一体化结构上投入过多定制加工时间。

# 10. 急停与安全硬件布局

## 10.1 急停按钮位置

急停按钮安装在**车体顶部中后部**，允许明显凸起。
这是为了保证操作者在跟车、侧边跟随或临近危险区域时，能够第一时间单手拍下急停按钮。

同时，为兼顾搬运时不易误触，建议在按钮周围增加**简易保护圈 / 护栏式结构**，但不能影响快速按下操作。

## 10.2 急停功能边界

根据计算通信架构与电源架构，硬件急停通过**纯硬件电路直接切断电机驱动供电**，不经过 Orin NX、STM32、ROS2 或 UART 链路。

这意味着：

- 急停按下后，动力链路立即失能；
- Orin NX、STM32、Mid360、D435 等计算与感知设备保持上电；
- 保留日志记录、状态监控、异常诊断和优雅关闭能力；

这一设计与迭代一安全机制“硬件急停 + 安全节点 + 禁区约束”的路线一致，且满足后续 P6 安全专项测试中对急停响应和异常停车可靠性的要求。

# 11. 模块间主要硬件链路

本节只定义硬件层面的主链路和接口关系，不展开具体引脚、线径、端子型号和接插件规格，这些内容由后续接线图、BOM 和电气实现阶段逐步细化。

## 11.1 感知链路

- Mid360 → Orin NX（USB / 以太网）
- D435 → Orin NX（USB）

## 11.2 控制链路

- Orin NX ↔ STM32（UART）
- STM32 → 电机驱动板（PWM）
- 编码器 → STM32（定时器 / 计数输入）

## 11.3 人工接管链路

- 飞智黑武士2 → 2.4G USB 接收器 → Orin NX → teleop / 手动控制节点 → `cmd_vel` → 底盘桥接 → STM32 → 驱动板。

## 11.4 安全链路

- 外部操作者 → 急停按钮 → 纯硬件电路 → 切断电机驱动供电。

## 11.5 电源相关硬件链路（与电源架构对应）

- 主电池 → 主入口 / 主保护 / 总开关；
- 主电池 → 动力支路 → 驱动板；
- 主电池 → 计算支路 → Orin NX；
- 主电池 → 传感器 / 控制支路 → STM32 / Mid360 / D435 / 补光控制。

# 12. 与软件架构的映射关系

项目0软件架构采用单一主 ROS2 workspace，在 Orin NX 上统一承载主业务软件，并通过明确的话题、服务、动作接口解耦；STM32 不运行 ROS2 主业务，仅通过底盘桥接接入 ROS2 世界。

对应到硬件层的映射关系如下：

- **执行模块**：STM32 + 电机驱动板 + 编码器 + 底盘本体；
- **感知模块**：Mid360 + D435；
- **主计算模块**：Orin NX；
- **安全保障模块**：硬件急停 + STM32 底层保护 + Orin NX 上层安全监控；
- **系统管理与数据记录模块**：运行在 Orin NX 上，因此 Orin NX 的供电稳定性、接口可达性和安装维护性在硬件架构中具有高优先级。

这说明硬件架构不是独立的机械堆叠，而是必须服务于软件主链路“感知输入—空间认知—行动规划—任务决策—底盘控制”和辅助链路“安全、管理、记录”的运行需求。

# 13. 与电源架构的接口关系

本硬件架构直接受 `power_architecture.md` 约束，并为其提供物理落位基础。

需要保持一致的接口边界包括：

1. **单主电池统一供能**：电池物理位置需适应三支路分配；
2. **动力支路独立**：驱动板位置需方便急停前后布线；
3. **Orin NX 独立高优先级供电**：上层应为 Orin NX 与其稳压支路预留清晰安装与维护空间；
4. **STM32 / Mid360 / D435 / 补光属于传感器/控制支路**：上层与前向感知区需便于该支路走线；
5. **电池电压监测**：STM32 与电源采样线需要可达；

换言之，电源架构定义“谁给谁供电”，硬件架构定义“这些供电对象物理上放在哪里、线往哪走、怎样避免互相妨碍”。

# 16. 当前结论

基于当前已知信息，项目0迭代一硬件架构已经可以在**主布局与系统边界层面形成正式文档**。

其核心结论如下：

1. 项目0在 P1 阶段正式建立 `docs/02_architecture/hardware_architecture.md` 作为硬件架构基线文档。
2. 平台采用现成底盘板 / 通用板材方案，双层板 + 立柱连接，优先快速装起来。
3. 底盘为四轮差速平台，主计算在 Orin NX，实时执行在 STM32。
4. Mid360 位于顶部接近几何中心，D435 与补光位于前向感知区，急停位于顶部中后部。
5. 人工接管采用飞智黑武士2 + 2.4G USB 接收器接 Orin NX 的上位机 teleop 方案；最高优先级安全仍由纯硬件急停保证。

7. 本文档已足以作为 P2 机械组装、传感器支架安装、人工接管链路建立和急停测试的输入基线。
