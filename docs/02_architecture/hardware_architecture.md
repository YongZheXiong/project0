## 1. 文档定位

本文档属于项目0在 P1 阶段的系统架构设计输出物之一，放置在：

```text
docs/02_architecture/hardware_architecture.md
```

本文档与以下架构文档并列管理：

```text

docs/02_architecture/compute_comm_architecture.md
```

本文档用于承接系统总架构中未展开的硬件实现层内容，定义项目0迭代一小车的硬件基线方案。

本文档的目标不是替代实物装配图、BOM、接线图、原理图或 CAD 图，而是为迭代一平台提供一个真实可运行、可维护、可扩展、可联调、可验收的硬件架构基线。

本文档重点回答以下问题：

1. 小车由哪些主要硬件模块组成；
2. 这些硬件模块在整车中分别承担什么职责；
3. 主电源、动力、计算、传感器/控制、安全急停之间如何分层；
4. Orin NX、STM32、传感器、电驱、电机之间如何形成硬件链路；
5. 哪些内容属于当前 P1 架构级确定项，哪些留到后续 BOM、接线图和 P2-P3 实物联调中细化。

## 3. 硬件架构设计目标

项目0硬件架构在迭代一阶段应满足以下目标。

### 3.1 支撑迭代一最小闭环成立

硬件架构必须优先服务于迭代一平台主干能力成立，包括：

- 底盘可控运动；
- 编码器反馈与基础里程计；
- Orin NX 与 STM32 稳定通信；
- Mid360 接入与建图定位；
- D435 接入与前向辅助感知；
- 硬件急停与人工接管；
- 主电池电压检测与动力支路电流检测；
- 补光模块接入与可控开关；
- 后续数据记录、系统管理、安全状态上报。

### 3.2 保持单人可落地

项目0由单人 solo 完成，因此硬件架构必须避免在 P1-P2 阶段引入过高机械复杂度或电气复杂度。

硬件路线优先采用：

- 现成底盘板 / 通用板材；
- 双层板结构；
- 常见 DC-DC 模块；
- 常见电驱板；
- 可购买或简单制作的支架；
- 可逐步调试、可逐步替换的模块化结构。

### 3.3 保持安全底线

项目0运行于真实教学楼走廊环境，存在楼梯口、天井边缘、电梯口、夜间弱光、排水槽等风险区域。

硬件架构必须保留以下安全底线：

- 硬件急停独立于软件链路；
- 急停只切断动力支路，不切断 Orin NX、STM32、传感器和日志记录能力；
- STM32 保持底层限幅、通信超时停车和底层状态上传能力；
- Orin NX 保持上层安全管理、系统管理和数据记录能力；
- 主电池电压和动力支路电流进入底层状态监测链路。

### 3.4 支撑后续成果表达

硬件架构应便于后续展示、解释、拍照、录制视频和写入报告。

因此，整车布局应尽量做到：

- 模块边界清楚；
- 走线方向可解释；
- 上下层职责明确；
- 传感器安装位置合理；
- 安全链路可被单独说明；
- 后续接线图、BOM、照片和测试记录能够与本文档对应。

## 5. 已确认的硬件决策

基于当前阶段已经确认的方案，项目0迭代一硬件架构采用以下硬件决策。

### 5.1 底盘形式

底盘采用四轮驱动底盘。

系统控制抽象采用左右两侧等效差速控制，即：

- 左侧电机组；
- 右侧电机组；
- 上层仍使用线速度与角速度形式表达运动意图；
- 底层由 STM32 和电驱板负责转换为实际电机控制。

迭代一不引入复杂四轮独立高级控制。

### 5.2 车体结构

车体采用现成底盘板 / 通用板材方案。

车体结构采用双层板：

- 下层：动力与供电；
- 上层：计算、控制与调试；
- 顶部 / 前向：感知、补光、急停触达。

双层板之间采用立柱连接。

当前阶段优先目标是快速装起来，不优先追求复杂定制机械结构。

### 5.3 主计算平台

主计算平台采用：

```text
Jetson Orin NX 16GB
```

Orin NX 作为 ROS2 主业务承载平台，负责：

- 传感器接入；
- 底盘桥接；
- 定位与建图；
- 地图管理；
- 导航与避障；
- 语义导航；
- 任务管理；
- 安全管理；
- 系统管理；
- 数据记录与复盘。

### 5.4 底层控制平台

底层控制平台采用：

```text
STM32
```

STM32 负责：

- 接收 Orin NX 下发的速度指令；
- 输出 PWM / DIR / EN 控制电机驱动板；
- 读取四个电机编码器；
- 进行底层速度闭环；
- 采集主电池电压；
- 采集动力支路电流；
- 读取急停状态；
- 读取电驱板故障状态；
- 实现通信超时停车；
- 上传底盘状态、编码器、轮速、电压、电流、急停状态和异常标志。

### 5.5 电机驱动方案

迭代一采用：

```text
两块双路电机驱动板
```

两块电驱板分别服务左右两侧或四个电机通道。

电驱板与 STM32 的基本接口为：

```text
STM32 → 电驱板：PWM / DIR / EN
电驱板 → STM32：FAULT / DIAG，若驱动板支持
```

电机驱动板选型时优先考虑：

- 支持当前主电池电压范围；
- 连续电流和峰值电流有足够余量；
- 支持 PWM / DIR 控制；
- 支持 EN / ENABLE 使能；
- 支持过流保护；
- 支持过温保护；
- 支持欠压保护；
- 支持 FAULT / DIAG 故障输出；
- 便于安装在下层动力区；
- 便于散热和接线。

### 5.6 电机与编码器

四个电机分别为：

- 左前电机；
- 左后电机；
- 右前电机；
- 右后电机。

四个电机均应具备编码器反馈。

编码器反馈进入 STM32，用于：

- 轮速计算；
- 速度闭环；
- 基础里程计；
- 底盘状态反馈；
- 后续导航和定位辅助；
- P2 底盘调试与验收。

### 5.7 主传感器

主传感器采用：

```text
Livox Mid360
```

Mid360 安装于车体顶部接近几何中心的位置，优先保证视场，允许明显高出上层板。

Mid360 是迭代一定位建图主传感器。

### 5.8 辅助视觉传感器

辅助视觉传感器采用：

```text
Intel RealSense D435
```

D435 前向安装，作为：

- 前向 RGB-D 辅助感知；
- 调试观察数据源；
- 数据记录来源；
- 后续迭代二多传感器融合增强预留基础。

迭代一阶段 D435 不作为 FAST-LIO2 主融合输入。

### 5.9 补光模块

补光模块布置在 D435 附近，用于弱光视觉支撑。

补光模块通过补光控制模块接入系统。

迭代一至少支持软件手动开关，后续可预留由视觉感知流程自动触发的接口。

### 5.10 人工接管方案

人工接管方案采用：

```text
飞智黑武士2手柄 + 2.4G USB 接收器
```

链路为：

```text
飞智黑武士2手柄
→ 2.4G USB 接收器
→ Orin NX
→ teleop / 手动控制节点
→ cmd_vel
→ 底盘桥接
→ UART
→ STM32
→ 电机驱动板
→ 电机
```

2.4G USB 接收器作为 Orin NX 的 USB 外设，由 Orin NX 的 USB 口接入，不作为独立低压负载直接挂到传感器/控制支路 DC-DC 下。

### 5.11 硬件急停方案

急停按钮布置于车体顶部中后部，明显凸起，便于人跟车时快速拍下。

硬件急停回路为纯硬件链路：

```text
动力支路
→ 硬件急停回路
→ 左电驱板 / 右电驱板
```

硬件急停回路的作用是切断电机驱动板供电。

急停动作不经过 Orin NX、STM32、ROS2 或 UART。

急停只切断动力支路，不切断：

- Orin NX；
- STM32；
- Mid360；
- D435；
- 补光控制；
- 电池电压检测；
- 数据记录能力；
- 系统状态上报能力。

急停状态应作为检测信号回传至 STM32，再通过 UART 上报 Orin NX，用于系统状态显示、日志记录、安全事件记录和后续测试验收。

### 5.12 电源监测与动力监测

迭代一确定实现以下采样 / 检测：

1. 主电池电压检测；
2. 电驱 / 动力支路总电流检测；
3. 四路编码器反馈；
4. 急停状态检测；
5. 电驱板 FAULT / DIAG 检测，若驱动板支持；
6. 电驱板 EN 使能控制，若驱动板支持。

迭代一不做以下内容：

1. 单个电机电流检测；
2. 高精度 SOC；
3. 剩余续航预测；
4. 计算支路 DC-DC 在线电压采样；
5. 传感器/控制支路 DC-DC 在线电压采样；
6. 低压支路电流在线采样；
7. 复杂 EMI 监测；
8. 多级冗余供电监测。

## 6. 整车分层布局原则

双层底盘采用：

```text
下层动力与供电
上层计算与控制
顶部感知与安全触达
```

的三层布局思路。

### 6.1 下层动力与供电

下层优先放置：

- 主电池；
- 主电源入口；
- 主保护器件；
- 总开关；
- 电源分配；
- 硬件急停回路相关动力走线；
- 两块双路电机驱动板；
- 动力支路电流检测模块；
- 计算支路 DC-DC；
- 传感器/控制支路 DC-DC。

下层布局原则：

1. 重物尽量靠下；
2. 高电流动力线尽量短；
3. 电驱板靠近电池和电机线束出口；
4. 电驱板保留自然散热空间；
5. 不让高电流动力线穿越上层计算区；
6. 为后续保险、端子、接插件和接线图细化预留空间。

### 6.2 上层计算与控制

上层优先放置：

- Orin NX；
- STM32；
- USB 接收器；
- 调试接口；
- UART 连接；
- 传感器数据线入口；
- 电源采样线入口。

上层布局原则：

1. Orin NX 方便连接 Mid360 和 D435；
2. Orin NX 散热不被遮挡；
3. STM32 靠近底层控制与采样线入口；
4. UART、USB、电源线、调试线应便于插拔；
5. 上层应便于单人调试和问题定位。

### 6.3 顶部与前向感知区域

顶部 / 前向区域放置：

- Mid360；
- D435；
- 补光模块；
- 急停按钮。

顶部 / 前向布局原则：

1. Mid360 优先保证视场；
2. D435 前向无遮挡；
3. 补光模块靠近 D435；
4. 急停按钮明显、可触达；
5. 急停按钮需要兼顾防误触；
6. 传感器支架不在 P1 阶段做复杂定制。

## 7. 供电拓扑原则

整车采用单主电池统一供能。

主电池经主电源入口、主保护器件和总开关后进入整车电源分配。

整车供电分为三条主支路：

```text
动力支路
计算支路
传感器/控制支路
```

### 7.1 动力支路

动力支路为：

```text
主电池
→ 主电源入口 / 主保护 / 总开关
→ 电源分配
→ 动力支路
→ 硬件急停回路
→ 动力支路电流检测
→ 左右两块电机驱动板
→ 四个电机
```

动力支路受硬件急停控制。

### 7.2 计算支路

计算支路为：

```text
主电池
→ 主电源入口 / 主保护 / 总开关
→ 电源分配
→ 计算支路 DC-DC
→ Orin NX
```

计算支路不受硬件急停切断。

### 7.3 传感器/控制支路

传感器/控制支路为：

```text
主电池
→ 主电源入口 / 主保护 / 总开关
→ 电源分配
→ 传感器/控制支路 DC-DC
→ STM32 / Mid360 / D435 / 补光控制模块
```

传感器/控制支路不受硬件急停切断。

### 7.4 2.4G USB 接收器供电边界

2.4G USB 接收器通过 Orin NX 的 USB 口接入并供电，不作为传感器/控制支路 DC-DC 的独立负载。

### 7.5 主电池电压检测

主电池电压检测链路为：

```text
主电池
→ 主电池电压检测模块
→ STM32 ADC
→ UART
→ Orin NX
```

该链路用于：

- 低电量告警；
- 运行准入；
- 低电量保护；
- 系统状态显示；
- 数据记录；
- 后续 P6 安全验收。

## 8. 计算、控制与通信链路

Orin NX 与 STM32 之间采用 UART 双向通信。

### 8.1 Orin NX 到 STM32

Orin NX 到 STM32 的下行逻辑流包括：

- 速度指令；
- 停车指令；
- 控制模式；
- 软件安全停车意图；
- 人工接管指令。

### 8.2 STM32 到 Orin NX

STM32 到 Orin NX 的上行逻辑流包括：

- 编码器数据；
- 轮速；
- 底盘状态；
- 电池电压；
- 动力支路电流；
- 急停状态；
- 电驱故障状态；
- 通信状态；
- 异常标志。

### 8.3 STM32 到电驱板

STM32 到电驱板的控制信号包括：

```text
PWM
DIR
EN / ENABLE
```

### 8.4 电驱板到 STM32

电驱板到 STM32 的反馈信号包括：

```text
FAULT / DIAG
```

是否接入取决于最终电驱板硬件能力。

## 9. 感知硬件链路

### 9.1 Mid360

Mid360 接入 Orin NX。

物理链路为：

```text
Mid360 → USB / 以太网 → Orin NX
```

Mid360 用于迭代一主定位建图链路。

### 9.2 D435

D435 接入 Orin NX。

物理链路为：

```text
D435 → USB → Orin NX
```

D435 用于前向辅助视觉 / 深度感知、调试和数据记录。

### 9.3 补光模块

补光模块通过补光控制模块接入传感器/控制支路。

控制方式为：

```text
Orin NX / STM32 控制意图
→ 补光控制模块
→ 补光模块
```

迭代一至少支持可控开关。

## 10. 人工接管链路

人工接管硬件链路为：

```text
飞智黑武士2手柄
→ 2.4G USB 接收器
→ Orin NX
→ teleop / 手动控制节点
→ cmd_vel
→ 底盘桥接
→ UART
→ STM32
→ 电驱板
→ 电机
```

人工接管不替代硬件急停。

硬件急停仍是最高优先级的物理安全手段。

## 11. 安全硬件链路

项目0迭代一硬件安全链路由三层组成：

1. 纯硬件急停；
2. STM32 底层保护；
3. Orin NX 上层安全监控。

### 11.1 纯硬件急停

硬件急停独立于软件，直接切断动力支路。

### 11.2 STM32 底层保护

STM32 负责：

- 速度限幅；
- 通信超时停车；
- EN 使能控制；
- 编码器反馈读取；
- 电压、电流、急停状态和电驱故障状态采集；
- 底层状态上传。

### 11.3 Orin NX 上层安全监控

Orin NX 负责：

- 安全状态聚合；
- 系统模式管理；
- 软件侧安全停车；
- 日志记录；
- 运行监控；
- 后续 P4-P6 安全联调。

## 16. 当前结论

基于当前已确认信息，项目0迭代一硬件架构在模块级已经形成可执行基线。

当前结论如下：

1. 项目0在 P1 阶段正式维护 `docs/02_architecture/hardware_architecture.md` 作为硬件架构基线文档。
2. 整车采用四轮差速底盘与双层板车体结构。
3. 主计算平台为 Orin NX，底层控制平台为 STM32。
4. 整车采用单主电池统一供能。
5. 电源分为动力支路、计算支路、传感器/控制支路。
6. 动力支路经硬件急停回路后给左右两块双路电驱板供电。
7. 两块双路电驱板驱动四个电机。
8. 四个电机编码器反馈进入 STM32。
9. 主电池电压检测、动力支路电流检测、急停状态和电驱故障状态进入 STM32。
10. STM32 与 Orin NX 通过 UART 双向通信。
11. Mid360 和 D435 接入 Orin NX。
12. 2.4G USB 接收器作为 Orin NX 的 USB 外设接入。
13. 补光模块接入系统控制，服务弱光视觉支撑。
14. 硬件急停只切断动力支路，不切断计算、控制、感知和记录支路。
15. 当前图纸与本文档用于指导 P1.3 采购清单、BOM、P2 机械组装和后续接线图细化。
