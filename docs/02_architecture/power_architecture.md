# Power Architecture

# 项目0 迭代一电源架构

## 1. 文档定位

本文档属于 P1 阶段系统架构设计输出之一，用于明确项目0迭代一平台的电源路径、电压转换方案、电源保护边界与电源监测机制，为后续硬件接线、BOM、采购、P2 机械与电气组装、P6 系统级验收提供依据。

P1 已明确要求输出电源架构，其核心内容包括：**电池→各设备供电路径、电压转换方案、电源保护**。

建议文件放置位置为 `docs/02_architecture/power_architecture.md`，与系统总架构、计算通信架构、硬件架构、软件架构并列管理。

---

## 2. 架构目标

迭代一电源架构的目标不是一次性做成复杂电源系统，而是支撑平台最小闭环正式成立，并满足后续调试、验收和扩展的基础需求。

本电源架构应满足以下目标：

1. 为四轮差速底盘、Orin NX、STM32、Mid360、D435、补光模块与遥控接收链路提供清晰供电关系。
2. 满足硬件急停的独立安全边界：**按下急停后，动力立即切断，但计算与感知链路不断电**，从而保留状态监控、日志记录和后续处置能力。
3. 支撑系统管理能力中的**电量/供电状态检查、运行期电源状态监控、低电量保护**。
4. 保持架构简单、可维护、适合单人开发，不为了未来扩展而在迭代一引入过重电源复杂度。

---

## 3. 电源架构设计约束

### 3.1 系统计算与执行边界

项目0采用“**单主计算平台 + 单底层控制平台 + 外设接入**”结构，Orin NX 是唯一主计算与主控平台，STM32 是唯一底层执行控制平台。

Orin NX 承担感知接入、定位建图、导航避障、语义导航、任务调度、安全监测、系统管理与数据记录等主业务，因此其供电必须作为高优先级稳定支路单独考虑。

STM32 承担电机驱动、编码器读取、里程计初步计算、底盘安全保护、接收上位机速度指令、上传底层状态等职责，因此它与电机驱动链路、电量监测链路直接相关。

### 3.2 传感器与补光约束

迭代一关键传感器包括 Mid360 与 D435，其中 Mid360 是 FAST-LIO2 的主感知输入来源，D435 在迭代一中作为前向辅助感知、调试与记录数据源，不参与 FAST-LIO2 正式融合。

系统还需具备补光支撑能力，补光模块应支持手动启用或由视觉感知流程触发，并具备开关控制能力，以在弱光场景下支撑视觉感知。

### 3.3 安全与管理约束

硬件急停按钮通过**纯硬件电路直接切断电机驱动供电**，不经过软件链路。

系统管理能力要求在启动时检查电量或供电状态是否满足运行条件，并在运行过程中监控电源状态。

P6 系统集成与验收要求补全**电源监控与低电量保护**能力，并在异常场景联调中覆盖低电量场景。

## 4. 电源架构总体原则

### 4.1 主电池统一供能
整车采用单主电池作为统一能源入口，再分支给动力、计算、传感器/控制等子系统供电，以保持供电关系清晰。

### 4.2 动力与计算分离
动力支路与计算支路必须电气分离，避免电机负载波动直接影响 Orin NX 主计算平台稳定性。

### 4.3 急停只切动力支路
急停仅切断电机驱动供电，不切断 Orin NX、STM32、Mid360、D435 等计算、控制、感知支路，以保留状态监控、故障记录、人工诊断和优雅关闭能力。

### 4.4 低压侧按职责分层
低压侧至少拆分为：
- 计算支路
- 传感器/控制支路

而不是把所有低压设备挂在同一低压模块下，以提高问题隔离性、维护性与可观测性。

### 4.5 迭代一先做基础监测与保护
迭代一先实现：
- 电池电压监测
- 低电量告警
- 低电量保护触发接口
- 供电关系清晰化

更高级的电量估算、剩余续航预测、复杂功耗优化不纳入当前阶段。

## 5. 供电对象清单

本架构覆盖以下主要供电对象：

### 5.1 动力侧
- 四轮底盘电机驱动板
- 四个 GB37-520 电机（由驱动板输出到电机）

### 5.2 计算侧
- Jetson Orin NX 16GB

### 5.3 传感器/控制侧
- STM32 控制板
- Livox Mid-360
- Intel RealSense D435
- 遥控接收机
- 补光模块控制支路

### 5.4 监测侧
- 主电池电压采样链路（建议由 STM32 负责采样与上报）

## 6. 电源分层与供电路径

## 6.1 总体拓扑

整车电源拓扑定义为：

**主电池 → 主电源入口 → 分为动力支路 / 计算支路 / 传感器控制支路 → 各设备**。

其中，动力支路与低压支路并联分出；低压支路内部再拆分为计算支路和传感器/控制支路。

---

## 7. 动力支路设计

### 7.1 动力支路路径

动力支路采用如下路径：

**主电池 → 主保护器件/总开关 → 硬件急停回路 → 电机驱动板 → 四个电机**。

### 7.2 动力支路电压策略

迭代一采用 **3S 主电池直接供给电机驱动板** 的方案，不在电池与驱动板之间额外引入一层大功率中间稳压级。
该方案符合迭代一“先做成、先闭环、先保证稳定运动和可靠停车”的策略，也符合 STM32 通过 PWM 驱动电机驱动板完成底盘运动控制的实现路径。

### 7.3 动力支路安全边界

动力支路是硬件急停的直接作用对象。
按下急停按钮后，应立即切断电机驱动板供电，从而实现电机失能与整车停止。

此外，底盘安全保护还包括 STM32 固件中的速度限幅与通信超时自动停车，这是硬件急停之外的第二层执行安全保障。

---

## 8. 计算支路设计

### 8.1 计算支路路径

计算支路采用如下路径：

**主电池 → 独立 DC-DC 稳压模块 → Orin NX**。

### 8.2 计算支路设计原则

Orin NX 是系统唯一主计算与主控平台，承担感知、定位、导航、任务、安全、系统管理、数据记录等全部上层关键能力，因此其供电应被视为**高优先级稳定支路**。

该支路应满足：
- 与动力支路电气隔离
- 不受急停切断
- 供电稳定，优先保证主计算平台不断电
- 能支撑系统监控、日志记录、模式切换与优雅关闭流程。

### 8.3 设计意图

保留 Orin NX 持续供电，能够在急停、通信中断、导航失败、低电量等异常场景下继续记录状态、输出异常等级、执行系统管理策略，并为 P6 的全链路联调与验收提供支撑。

---

## 9. 传感器/控制支路设计

### 9.1 支路路径

传感器/控制支路采用如下路径：

**主电池 → 独立 DC-DC 稳压模块 → STM32 / Mid360 / D435 / 遥控接收机 / 补光控制支路**。

### 9.2 支路职责

该支路承载以下设备：

- STM32：底层控制、底层状态上传、电量采样上报、接收人工接管输入。
- Mid360：点云与内置 IMU 数据输入，直接支撑 FAST-LIO2 主路径。
- D435：前向辅助视觉/深度感知、调试可视化与数据记录。
- 遥控接收机：人工接管硬件入口，经 STM32 参与控制权切换。
- 补光模块控制通道：支撑弱光视觉感知与补光开关控制。

### 9.3 不与 Orin 共用单模块的原因

尽管这些设备都属于低压侧，但传感器/控制支路与计算支路分开，更符合职责边界清晰、故障隔离、调试维护简单的工程原则，也更适合单人开发中的快速定位问题。

---

## 10. 补光支路设计

### 10.1 功能定位

补光模块属于迭代一明确要求接入的感知支撑能力之一，其目标是在弱光场景下支撑 D435 的基本视觉感知能力。

### 10.2 支路关系

补光模块不单独从主电池直接取电，而是作为**传感器/控制支路下的受控负载**存在，以保持供电关系清晰并便于统一管理。

### 10.3 控制要求

补光模块需具备：
- 手动启用能力
- 系统控制开关能力
- 与视觉感知流程协同的基础接口能力

这与补光支撑能力的要求保持一致。

---

## 11. 电量监测与低电量保护设计

### 11.1 电量监测定位

系统必须具备：
- 电池电量监测能力
- 低电量告警能力
- 低电量保护策略
- 供电路径清晰管理能力

启动时，系统管理链路还应检查电量或供电状态是否满足运行条件。

### 11.2 监测实现边界

迭代一建议采用**主电池电压监测**作为基础实现路径，由 STM32 负责采样，再通过 UART 上报给 Orin NX，由系统管理与安全管理逻辑消费该状态。

这符合：
- STM32 已承担底层状态上传职责
- Orin NX 统一承担系统管理、运行监控与数据记录职责

### 11.3 低电量处理策略

迭代一阶段建议定义至少三级电量状态：

1. **正常**
2. **低电量告警**
3. **低电量保护**

低电量保护策略至少应支持：
- 禁止进入新的自主任务
- 触发系统告警
- 必要时触发安全停车/受限运行

这与系统管理能力、异常分级管理能力和 P6 电源监控与低电量保护要求一致。

---

## 12. 急停与供电边界

### 12.1 急停作用范围

硬件急停按钮只切断**电机驱动供电**，不切断：
- Orin NX
- STM32
- Mid360
- D435
- 遥控接收机
- 电量监测与日志记录支路

### 12.2 这样设计的原因

该设计满足以下要求：

1. 硬件急停直接作用于最危险的动力执行链路，安全优先级最高。
2. 保留系统管理、自检、状态输出、异常记录和优雅关闭能力，避免“直接断电式关闭”。
3. 符合数据记录链路应覆盖全过程、不参与控制优先级竞争但全程伴随的系统设计原则。
4. 便于 P6 异常场景联调与安全专项测试中的急停响应、异常停车可靠性验证。

---

## 13. 电源保护原则

迭代一电源保护先遵循基础清晰原则，不追求过度复杂。

### 13.1 主入口保护
主电池输出后应设置主保护器件与总开关，作为整车上电入口的第一道保护。

### 13.2 动力侧保护
动力支路应具备：
- 硬件急停切断能力
- STM32 通信超时自动停车
- STM32 速度限幅保护

驱动板若支持过流/过温保护，可作为增强项接入；若硬件能力不足，迭代一不强制复杂实现。

### 13.3 低压侧保护
低压侧至少通过独立稳压模块实现：
- 计算支路保护
- 传感器/控制支路保护

以避免单点低压故障扩散影响全部设备。

## 14. 电源架构文字图

```text
主电池(3S)
│
├─ 主电源入口 / 主保护 / 总开关
│
├─ 动力支路
│   └─ 硬件急停回路
│       └─ 电机驱动板
│           └─ 四个驱动电机
│
├─ 计算支路
│   └─ 独立DC-DC稳压
│       └─ Orin NX
│
└─ 传感器/控制支路
    └─ 独立DC-DC稳压
        ├─ STM32
        ├─ Mid360
        ├─ D435
        ├─ 遥控接收机
        └─ 补光控制支路
            └─ 补光模块

主电池电压采样：
主电池 ─→ STM32采样 ─→ UART上报 ─→ Orin NX系统管理/安全管理
```

该拓扑符合 P1 对电源架构的定义要求，也与计算通信架构、系统管理能力、安全机制和电源管理能力保持一致。

## 15. 与软件/系统管理链路的接口关系

电源架构不是孤立硬件设计，它需服务于软件与系统管理链路。

### 15.1 系统管理链路
Orin NX 上的系统管理链路负责：
- 启停管理
- 状态管理
- 自检
- 配置管理
- 运行监控

因此，电源架构必须向系统管理提供至少以下状态输入：
- 主电池电压状态
- 低电量等级
- 急停状态
- 底层供电相关异常状态

### 15.2 安全链路
安全保障链路由：
- 纯硬件急停
- STM32 底层保护
- Orin NX 上层安全监控

三层构成。

因此电源架构中的急停与低电量保护，应被视为安全链路的一部分，而不只是“硬件供电细节”。

## 16. 当前版本不细化内容

为保持迭代一范围收敛，以下内容在本版架构中暂不定死：

1. 各 DC-DC 的具体输出电压数值与型号
2. 线径、连接器、端子、保险丝规格
3. 驱动板具体型号与最大电流余量
4. 电池 SOC 高精度估算方案
5. 复杂电源滤波、EMI 细节与多级冗余设计
6. 复杂功耗优化与续航预测模型

这些内容应在 BOM、接线图、采购清单和后续电气实现阶段逐步细化，而不在当前版本过度展开。

## 18. 当前结论

基于当前已知信息，项目0迭代一电源架构已经可以被**主拓扑层面完全约束并形成正式文档**。

其核心结论为：

1. 采用单主电池统一供能。
2. 电源分为动力支路、计算支路、传感器/控制支路三大部分。
3. 动力支路由主电池直供驱动板，急停只切动力支路。
4. Orin NX 独立高优先级稳压供电，不受急停切断。
5. STM32、Mid360、D435、遥控接收机、补光控制挂载于独立传感器/控制支路。
6. 迭代一正式实现电池电压监测、低电量告警和低电量保护基础能力。
7. 补光模块纳入系统控制，作为弱光视觉支撑链路的一部分。

该架构符合项目0迭代一“先平台成立、先闭环成立、先安全可控、再逐步增强”的总原则，也与后续 P2-P6 的执行、感知、系统管理、安全验收路径保持一致。
