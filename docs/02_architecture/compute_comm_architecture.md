## 2. 架构目标与原则

项目0是一个面向教学楼走廊真实场景、以真机闭环为验收标准的具身智能地面移动平台。计算与通信架构必须服务于这一定位，并与系统总架构保持一致。

### 2.1 单主计算平台

系统只有一个明确的主计算中心：**Orin NX**。

### 2.2 单底层执行控制平台

系统只有一个明确的底层执行控制中心：**STM32F407**。

### 2.3 上层集中、下层确定

- Orin NX 负责上层系统闭环（感知、定位、导航、任务、安全监测、系统管理、数据记录）
- STM32 负责底层实时闭环（电机控制、编码器读取、底盘安全保护）

这是整份文档最核心的职责划分原则。后续所有通信链路和节点落位均以此为基础。

### 2.4 先支撑迭代一，再承接迭代二

计算与通信架构必须先支撑迭代一平台成立，再在不推翻主从骨架的前提下承接迭代二增强。

### 2.5 外部操作终端不纳入正式计算节点

人工急停按钮和遥控器属于外部输入源，不作为系统内部 ROS2 节点。它们向系统注入信号，由系统内部的安全保障模块处理。

## 3. 计算实体划分

项目0采用"单主计算平台 + 单底层控制平台 + 外设接入"的结构。

### 3.1 Orin NX

系统唯一主计算与主控平台。

承担：

- 感知接入与管理（Mid360、D435、IMU 数据接收与发布）
- 定位、建图、地图管理
- 导航与避障
- 语义导航与任务调度
- 安全监测与上层安全处置
- 系统管理、自检、运行监控
- 数据记录与调试支撑

### 3.2 STM32F407

系统唯一底层执行控制平台。

承担：

- 电机驱动与 PID 速度闭环控制
- 编码器读取与里程计初步计算
- 底盘安全保护（速度限幅、通信超时停车、急停信号响应）
- 接收 Orin NX 下发的速度指令
- 向 Orin NX 上传编码器数据与底层状态

### 3.3 外设接入

以下设备直接接入 Orin NX：

| 设备 | 接入方式 | 角色 |
|------|---------|------|
| Livox Mid360 | USB/以太网 | 3D激光雷达，提供点云与内置IMU数据 |
| Intel RealSense D435 | USB | 深度相机，提供RGB与深度图像 |

> **D435 在迭代一中的角色说明**：D435 在迭代一中仅作为深度/视觉数据源接入系统，用于辅助感知、数据记录（rosbag2 录制）和调试可视化（RViz2 观测）。**D435 数据在迭代一中不流向 SLAM 节点组，不参与 FAST-LIO2 的定位融合**。迭代二前段才将 D435 视觉/深度信息融入定位系统。

以下设备接入 STM32：

| 设备 | 接入方式 | 角色 |
|------|---------|------|
| 编码器（左/右电机） | STM32 定时器硬件解码 | 提供轮速与里程计数据 |
| 电机驱动板 | STM32 PWM 输出 | 接收 PID 控制信号驱动电机 |

### 3.4 IMU 数据来源说明

迭代一中 IMU 数据的主路径为：**Mid360 内置 IMU，通过 Livox ROS2 驱动在 Orin NX 上直接发布** 。

备选路径：如需外置 IMU，则通过 STM32 采集后经 UART 上传至 Orin NX。**具体路径在硬件架构确认后锁定。** 在锁定之前，软件架构应同时考虑两种接入方式的兼容性。

## 4. 通信链路定义

### 4.1 Orin NX ↔ STM32 主从链路

| 项目 | 说明 |
|------|------|
| 物理链路 | UART 串口（USB转TTL或直连） |
| 协议 | 自定义二进制协议帧（帧头 +数据 + CRC校验 + 帧尾） |
| 下行数据（Orin NX → STM32） | 目标线速度、目标角速度 |
| 上行数据（STM32 → Orin NX） | 左右编码器计数、轮速、底层状态标志（正常/异常/急停） |
| 通信频率 | 建议 50-100Hz |
| 异常处理 | 超时检测（STM32 超过 200ms 未收到指令则自动停车）、校验失败丢帧 |

### 4.2 感知接入链路

| 传感器 | 物理接口 | 接入平台 | ROS2 驱动 | 数据流向 |
|--------|---------|---------|-----------|---------|
| Mid360 | USB/以太网 | Orin NX | livox_ros_driver2 | → SLAM节点组、导航节点组、数据记录节点组 |
| D435 | USB | Orin NX | Intel RealSense ROS2 Wrapper | → 数据记录节点组、调试可视化（迭代一）；迭代二增加→SLAM节点组 |
| IMU（主路径） | Mid360内置 | Orin NX（Livox驱动发布） | livox_ros_driver2 | → SLAM节点组 |

### 4.3 安全相关通信路径说明

安全链路是全系统最高优先级，其在通信层面的落位必须明确：

| 安全机制 | 通信路径 | 说明 |
|----------|---------|------|
| **硬件急停** | **纯硬件电路，不经过任何通信链路和软件** | 物理急停按钮直接切断电机驱动供电。这条路径独立于 UART、独立于 ROS2，是最后一道安全保障 |
| **遥控器人工接管** | 遥控器接收机 → STM32（接收 PWM 信号）→ STM32 判断是否切换为人工控制模式 | 遥控器信号直接进入 STM32，STM32 在底层固件中实现控制权切换逻辑：当检测到有效遥控信号时，忽略 Orin NX 下发的速度指令，转而执行遥控器指令 |
| **软件急停** | 安全监控节点（Orin NX ROS2）→ 底盘桥接节点（Orin NX ROS2）→ UART → STM32 停车 | 安全监控节点检测到异常（定位丢失/感知异常/导航卡死等）后，通过 ROS2 内部通信通知底盘桥接节点发送零速度或急停指令给 STM32 |

### 4.4 数据记录链路

rosbag2 和 ROS2 标准日志统一运行在 Orin NX 上。录制范围覆盖：

- 传感器原始数据话题（Mid360 点云、D435 RGB/深度、IMU）
- 里程计与定位结果话题
- 导航路径与速度话题
- 任务状态话题
- 安全事件与系统状态话题

### 4.5 通信链路总览图（文字描述）

```
外部操作者
  ├─ 硬件急停按钮 ──→ [纯硬件电路] ──→ 切断电机驱动供电
  └─ 遥控器 ──→ 接收机 ──→ STM32（控制权切换）

Orin NX（主计算平台）
  ├─ USB/以太网 ←── Mid360（点云 + IMU）
  ├─ USB ←── D435（RGB + 深度）
  ├─ UART ←→ STM32（速度指令↓ / 编码器+状态↑）
  └─ ROS2 内部通信（节点组之间）

STM32（底层控制平台）
  ├─ 定时器 ←── 编码器（左/右）
  ├─ PWM ──→ 电机驱动板
  ├─ UART ←→ Orin NX
  └─ PWM ←── 遥控器接收机
```

## 5. 系统总架构在计算平台上的落位

### 5.1 主链路落位

系统总架构定义的五层主链路在计算平台上的落位关系：

| 主链路层级 | 计算平台 | 说明 |
|-----------|---------|------|
| 执行层 | STM32 | 电机控制、编码器读取、底盘保护在 STM32 实时闭环中完成 |
| 感知层 | Orin NX | 传感器驱动与数据发布统一在 Orin NX 上 |
| 空间认知层 | Orin NX | FAST-LIO2 建图/定位运行在 Orin NX 上 |
| 行动规划层 | Orin NX | Nav2 导航框架运行在 Orin NX 上 |
| 任务决策层 | Orin NX | BehaviorTree.CPP + 语义导航逻辑运行在 Orin NX 上 |

### 5.2 辅助链路落位

| 辅助链路 | 计算平台 | 说明 |
|----------|---------|------|
| 安全保障链路 | STM32（底层保护）+ Orin NX（上层监测与处置）+ 纯硬件（急停按钮） | 三层安全：硬件急停不经软件；STM32 限幅和超时停车；Orin NX 安全监控节点 |
| 系统管理链路 | Orin NX | 启停管理、状态管理、自检、配置管理、运行监控统一在 Orin NX |
| 数据记录链路 | Orin NX | rosbag2 录制和日志输出统一在 Orin NX |

## 6. ROS2 节点组粗粒度规划

以下为 Orin NX 上运行的 ROS2 节点组规划。**本节只定义节点组及其系统角色，不定义组内各节点的具体拆分和职责——组内节点拆分由软件架构文档承接** 。

| 节点组 | 运行平台 | 系统角色 | 对应总架构模块 |
|--------|---------|---------|---------------|
| 感知接入节点组 | Orin NX | 运行传感器驱动，发布点云、图像、深度、IMU 等感知数据 | 感知模块 |
| SLAM 节点组 | Orin NX | 运行 FAST-LIO2，完成建图与定位 | 空间认知模块 |
| 导航节点组 | Orin NX | 运行 Nav2 框架，完成全局路径规划、局部避障、路径跟踪与代价地图管理 | 行动规划模块 |
| 任务决策节点组 | Orin NX | 运行 BehaviorTree.CPP 与语义导航逻辑，完成任务管理与语义目标转换 | 任务决策模块 |
| 底盘桥接节点组 | Orin NX | 负责 Orin NX 侧的 UART 通信处理，将 ROS2 速度指令转为串口数据下发，将串口上传数据转为 ROS2 话题发布 | 执行模块（Orin NX 侧桥接） |
| 安全监控节点组 | Orin NX | 监控定位/感知/导航/通信/电量等状态，检测异常，发布软件级安全指令 | 安全保障模块 |
| 系统管理节点组 | Orin NX | 管理系统模式、状态、自检、启动流程、参数配置与运行监控 | 系统管理模块 |
| 数据记录节点组 | Orin NX | 运行 rosbag2 录制与日志管理 | 数据记录与复盘模块 |

> STM32 上不运行 ROS2 节点，运行的是独立固件程序，负责电机控制、编码器读取、底盘安全保护和 UART 通信。

## 7. 数据流、控制流与状态流大方向

### 7.1 数据流

```
Mid360 ──点云+IMU──→ 感知接入节点组 ──→ SLAM节点组 ──→ 导航节点组
D435 ──RGB+深度──→ 感知接入节点组 ──→ 数据记录节点组（迭代一）
STM32 ──编码器+状态──→ 底盘桥接节点组 ──→ SLAM节点组 / 导航节点组
```

### 7.2 控制流

```
任务决策节点组 ──导航目标──→ 导航节点组 ──速度指令──→ 底盘桥接节点组 ──UART──→ STM32 ──PWM──→ 电机
```

### 7.3 状态流

```
各节点组 ──运行状态──→ 系统管理节点组（汇总系统状态）
各节点组 ──异常状态──→ 安全监控节点组（检测异常、触发保护）
各节点组 ──话题数据──→ 数据记录节点组（全程录制）
```

## 8. 迭代视角下的计算与通信架构稳定性

### 8.1 迭代一中的作用

在迭代一阶段，本架构支撑最小闭环平台成立：

- Orin NX 运行感知、SLAM、导航、任务、安全、管理、记录全部上层逻辑
- STM32 运行电机控制、编码器读取、底盘保护
- UART 连接上下位机
- Mid360 提供点云与 IMU 数据用于 FAST-LIO2
- D435 接入但仅用于记录和调试，不参与定位融合
- 遥控器信号接入 STM32 实现底层控制权切换

### 8.2 迭代二中的变化

迭代二在不改变主从骨架的前提下承接增强：

- **迭代二前段**（优先推进）：D435 数据流从"仅记录"扩展为"流向 SLAM 节点组参与视觉融合"；编码器里程计从"独立发布"扩展为"融入定位系统"；可能新增回环检测与退化处理逻辑——这些变化均在 Orin NX 内部的 ROS2 节点组之间发生，不改变 Orin NX ↔ STM32 的通信结构
- **迭代二后段**（在前段基本完成后推进）：任务决策节点组内部扩展语义导航深化逻辑——同样不改变计算与通信骨架

**结论：迭代二所有增强均为 Orin NX 内部节点组的能力增强或数据流扩展，主从架构骨架不变，通信链路不变。**

## 10. 总结

项目0计算与通信架构可以概括为：

> 项目0采用"Orin NX 作为唯一主计算与主控平台、STM32F407 作为唯一底层执行控制平台"的主从式计算与通信架构。感知接入、空间认知、导航避障、任务决策、安全监测、系统管理与数据记录统一收敛于 Orin NX；电机控制、编码器读取、底盘执行、底层安全保护与遥控器控制权切换统一收敛于 STM32。Mid360 与 D435 均由 Orin NX 接入（D435 在迭代一中仅用于记录和调试，不参与定位融合）。STM32 接收 Orin NX 的运动控制意图并上传底盘反馈，同时独立处理遥控器接管信号。硬件急停通过纯硬件电路直接切断电机驱动供电，不经过任何软件链路。由此形成"上层系统闭环在 Orin NX、底层实时闭环在 STM32、硬件急停独立于所有软件"的三层安全统一结构。该结构服务于迭代一平台成立，并在不改变主从骨架的前提下承接迭代二的多传感器融合增强、定位建图鲁棒性增强与语义导航深化。
```
